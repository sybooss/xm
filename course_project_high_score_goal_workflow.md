# 复杂软件系统实践高分 Goal Workflow

这个文件用于启动 Codex `/goal` 长期迭代模式，目标不是“把项目做完就停”，而是让 Codex 按课程评分、答辩展示和特色亮点持续优化整个项目。

本项目当前目标：

- 项目：电商退换货智能客服系统
- 路径：`D:\复制软件系统`
- 主栈：`Spring Boot + Vue 3 + MySQL + LangChain4j`
- 答辩形态：个人答辩，但先按完整项目做；最后再从完整项目中提炼个人贡献、核心代码讲解和答辩话术。
- 高分策略：基础要求必须完整可运行，特色功能要多、要能演示、要有证据，让老师看到“这个系统比普通课程设计更完整、更智能、更像真实项目”。
- 前端目标：页面必须有高级感，参考苹果官网式的克制、精致、通透、强层级和顺滑交互。不能停留在普通后台模板或粗糙表单页面。

## 最高原则

不要让 agent 只说“继续优化”。必须让它围绕真实评分和真实演示循环：

```text
课程要求/评分标准 -> 当前项目审计 -> 必做功能补齐 -> 特色功能池排序 -> 小步实现 -> 构建/冒烟/浏览器验证 -> 文档/报告/答辩材料同步 -> 自评 -> 继续下一轮
```

完整项目优先，个人答辩随后提炼。

特色功能可以多做，但每一个特色功能都必须满足：

- 代码里真的实现了。
- 前端或接口能演示。
- 数据库、API、文档和报告描述一致。
- 有验证命令、截图、日志、测试结果或演示脚本作为证据。
- 不牺牲系统稳定性，不绕过业务规则，不让 LangChain4j 直接替代后端业务逻辑。

每轮完成后必须自我迭代：站在老师、用户和答辩评委视角重新检查“还能不能更高级、更稳定、更有特色、更好讲”。只有当继续优化的收益已经很低，或剩余方向需要用户选择/外部资源时，才允许停止。

## 课程真实评分导向

如果没有新的老师要求，按当前课程评分标准优化：

| 模块 | 评分点 | 分值 | 高分策略 |
| --- | --- | ---: | --- |
| 开题报告 | 团队分工合理、方案设计可行 | 10 | 保证需求、架构、分工、计划和实际项目一致 |
| 开题报告 | 逻辑清晰、有创新性 | 10 | 把 AI 客服、知识库、工单、日志、权限等亮点写成系统化方案 |
| 结项答辩-团队 | 满足设计需求、运行顺畅 | 30 | 最高优先级：完整链路必须稳定演示 |
| 结项答辩-团队 | 程序设计规范、UI 友好 | 10 | RESTful、分层清晰、页面好看且好讲 |
| 结项答辩-个人 | 自身开发部分自述准确、全面 | 10 | 从完整项目中提炼个人负责模块、核心代码和改进点 |
| 结项答辩-个人 | 回答问题逻辑清晰、有见解 | 10 | 准备问答库、技术取舍、创新点、风险和改进方向 |
| 结项报告 | 内容详实、表达正确、逻辑清晰 | 10 | 报告必须对应真实系统，包含图表、截图、测试证据 |
| 结项报告 | 合理、可行的拓展建议 | 10 | 写出后续可扩展方向，最好与已实现特色功能呼应 |

自拟选题有难度系数影响，因此要用“工程完整度 + AI 特色 + 演示稳定性”抵消题目难度风险。

## 高分目标拆解

### 1. 必做功能先做到完整

先把项目作为一个完整系统打通，不要只做单个页面或单个接口。

必做基础能力：

- 登录、注册、角色权限和演示账号。
- 客服会话、消息发送、多轮追问和会话记录。
- 退货、换货、退款进度、物流异常、投诉升级等售后场景。
- 订单查询、售后记录、人工工单创建和处理。
- 知识库分类、文档管理、关键词检索和命中依据展示。
- AI 回复、LangChain4j 工具调用、本地规则兜底。
- 后台管理、日志追踪、AI 调用记录和演示数据。
- 数据库建表、初始化数据、README、接口文档、测试说明。

必做功能没有稳定前，不要把主要时间花在纯装饰功能上。

### 2. 特色功能要多，但要分层实现

老师说“有特色功能或者让人眼前一亮就能拿高分”，因此 `/goal` 迭代时要维护一个特色功能池，每轮优先做投入产出比最高的亮点。

建议至少形成 8 个以上可演示亮点，优先从这些方向选：

| 特色方向 | 可演示亮点 | 证据要求 |
| --- | --- | --- |
| AI 流式客服 | 回复过程逐字输出、处理进度实时展示 | 浏览器演示、接口冒烟 |
| RAG 知识增强 | 回复时展示命中的知识库依据 | 页面截图、知识库数据 |
| LangChain4j 工具调用 | AI 可调用订单查询、知识检索、工单创建 | 日志、接口结果、代码说明 |
| 本地规则兜底 | 模型不可用时仍能给出可靠业务回复 | 关闭或缺失模型 key 的冒烟测试 |
| 多轮上下文 | 用户追问“那退款呢”时能承接上一轮订单/场景 | 浏览器脚本、聊天记录 |
| 智能工单升级 | 投诉、异常物流、人工客服自动转工单 | 工单列表、数据库记录 |
| 权限控制 | 普通用户和管理员看到/能操作的内容不同 | 浏览器角色测试 |
| AI 调用日志 | 展示模型、耗时、请求摘要、结果状态 | 日志中心页面、数据库表 |
| 知识库命中分析 | 展示关键词、命中文档、命中次数 | 页面表格、接口返回 |
| 售后流程可视化 | 订单售后状态用步骤条、时间线或流程面板展示 | UI 截图、演示脚本 |
| 数据看板 | 工单数量、意图分布、热门问题、处理状态统计 | 页面截图、SQL/接口 |
| 模型切换 | 演示不同模型或本地网关配置切换 | 前端控件、后端配置 |
| Demo 一键脚本 | 一条命令跑后端、前端、数据库冒烟 | 脚本输出 |
| 答辩模式页面 | 专门给老师看的演示入口、亮点索引和测试账号 | 浏览器演示 |
| 苹果式高级前端 | 首屏、工作台、看板和详情页呈现克制高级、层级清楚、动效顺滑的视觉质感 | 浏览器截图、移动端/桌面检查 |
| 异常处理体验 | 网络失败、AI 失败、无订单、无知识命中都有友好提示 | 页面状态、测试用例 |
| 工程化亮点 | RESTful API、分层设计、统一响应、拦截器、CI/构建说明 | 代码引用、文档 |

特色功能不能只写在报告里。没有真实实现、无法截图、不能演示的内容，不算特色功能。

### 3. 完整项目优先于个人模块

虽然最后是个人答辩，但 `/goal` 要先把完整项目做到可运行、可讲、可演示。

个人答辩材料在后期从完整项目里提炼：

- 我的负责模块。
- 我解决的核心问题。
- 我写的关键接口、页面、数据库表或 AI 调用链路。
- 技术难点和解决方案。
- 为什么这样设计。
- 特色功能如何体现创新性。
- 如果老师追问，如何解释 LangChain4j、RAG、兜底规则、权限、日志和测试。

不要为了“个人部分”把系统拆散。老师看到完整项目运行顺畅，比只讲某个孤立模块更容易给高分。

## Agent 工作规则

当用户要求继续完善项目、冲高分、启动 `/goal`、做完整项目、做特色功能或准备答辩时，Codex 必须按本文件执行。

### 1. 从真实证据开始

每次长期迭代开始前，先检查真实项目，不要凭记忆或文件名猜测：

- 阅读 `AGENTS.md`。
- 阅读课程任务书、评分标准、README、报告、设计文档、接口文档、数据库脚本。
- 运行 `git status --short --branch`。
- 识别当前后端、前端、数据库、工具脚本和可运行命令。
- 检查当前已有功能、缺口、坏掉的地方和未提交文件。
- 如果使用历史记忆，只能作为线索，必须用当前文件和命令复核关键状态。

### 2. 维护 `.codex/course_goal.md`

非简单任务必须创建或更新 `.codex/course_goal.md`，作为长期迭代账本。

它至少包含：

- 当前目标和项目路径。
- 当前评分基线。
- 必做功能完成度。
- 特色功能池和优先级。
- 报告/答辩材料缺口。
- 每轮改动、验证命令、结果和剩余问题。
- 最终自评。

### 3. 每轮只做一个清晰增量

每轮选择最高收益的未完成项。

排序优先级：

1. 影响运行顺畅和完整演示的 bug。
2. 必做业务链路缺口。
3. 能让老师明显感知“智能、完整、工程化”的特色功能。
4. 报告、README、演示脚本和答辩材料同步。
5. UI 细节和视觉 polish。

不要一次塞太多不相关改动。特色功能可以多，但必须一轮一轮做，每轮都能验证。

### 4. 代码迭代规则

涉及代码时：

- 先读相关后端、前端、SQL、工具脚本。
- 保持主栈为 `Spring Boot + Vue 3 + MySQL + LangChain4j`。
- REST 接口优先使用资源名词和标准 HTTP 方法。
- AI 只能增强业务能力，不允许绕过 Spring Boot 业务服务直接写库。
- 必须保留本地规则兜底，模型不可用时系统仍可演示核心业务。
- 跨层功能要同步后端、前端、SQL、文档和测试。
- 不提交 `.env`、`output/`、`tmp/`、`web/dist/`、`server/target/`、`node_modules/` 等产物。

验证要求按修改范围选择：

- 后端：`mvn -q -DskipTests package`
- 前端：`npm run build`
- 数据库：执行或检查 `sql/schema.sql`，确认新增表/字段可访问
- 全链路：`tools/full-smoke-test.ps1`
- 浏览器：`npm run test:browser`
- 接口功能：补充 PowerShell/curl 冒烟测试

验证失败时，不要跳过。先定位原因，修复后重跑。

### 5. 前端高级感规则

前端页面是高分展示的第一印象。每次涉及页面、交互、展示、答辩演示时，必须按“高级产品页面”标准迭代。

视觉目标：

- 参考苹果官网的高级感：克制、干净、层级强、留白舒服、动效自然、重点突出。
- 不做廉价炫彩，不堆满屏渐变，不使用粗糙后台模板感。
- 页面要像一个真实可交付产品，而不是课程作业拼装页面。
- 首屏要让老师立刻知道项目主题、AI 特色和核心业务价值。
- 工作台、知识库、工单、日志、看板等页面要统一设计语言。

具体检查项：

- 字体层级清楚，标题、说明、数据、按钮不要混成一团。
- 留白、卡片、边框、阴影、色彩都要克制，不能乱。
- 重点数据和 AI 亮点要有视觉焦点。
- 表格、表单、聊天区、日志区要适合长时间使用。
- 动效要轻、顺、服务于理解，例如流式回复、状态变化、步骤推进。
- 移动端和桌面端都不能出现文字溢出、遮挡、按钮挤压或布局错乱。
- 页面文案要像产品文案，避免生硬解释“这里是某某功能”。
- 答辩演示页要有“亮点索引”，老师能快速看到系统特色。

每轮前端改动后必须做自我视觉审查：

1. 这页是否比普通 Element Plus 后台模板更高级？
2. 老师第一眼能否感受到项目完成度和 AI 特色？
3. 哪个区域还显得廉价、拥挤、空、乱或不好讲？
4. 还能不能通过布局、文案、动效、数据呈现或演示入口再提升？

如果仍有明显可优化点，并且不会破坏稳定性，就继续优化，不要急着停。

### 6. 文档、报告和答辩同步

每个已实现功能都要能在文档或报告里找到对应描述。

需要同步维护：

- `README.md`
- `docs/backend-api-design.md`
- `docs/frontend-project-doc.md`
- `docs/database-design.md`
- `docs/test-cases.md`
- `docs/demo-script.md`
- 开题/结项报告材料
- 答辩讲稿、Q&A、个人贡献说明

报告内容必须跟真实系统一致。不能编造测试结果、截图、老师反馈或不存在的功能。

### 7. 高分自评 Gate

每轮结束和最终交付前，都要按老师视角自评：

- 老师能否按 README 跑起来？
- 答辩现场能否稳定演示完整链路？
- 必做需求是否都有入口、数据和结果？
- 特色功能是否足够多，并且每个都能讲清楚价值？
- AI 相关亮点是否不是空壳，而是有日志、依据、工具调用或兜底逻辑？
- 前端页面是否有高级感，是否接近苹果官网那种克制、精致、顺滑、强视觉层级的产品感？
- 接口、数据库、前端、报告是否一致？
- 个人答辩是否能清楚说明自己做了什么、难点是什么、创新在哪里？
- 本轮结束前是否已经再次自我迭代，确认没有明显还能继续提高分数的改进点？
- 工作区是否只包含本轮相关文件？
- 是否完成必要的验证、提交和推送？

如果任何关键项不通过，继续修，不要输出完成承诺。

## Stop Conditions

Codex 只能用以下状态停止：

```text
PASS: 完整项目、特色功能、前端高级感、报告/答辩材料和验证证据都已完成，并且经过自我迭代审查后没有明显高收益优化点。
BLOCKED: 存在真实外部阻塞，必须给出准确错误、已尝试步骤和下一步人工动作。
NEEDS_USER_CHOICE: 存在多个高影响方向，擅自选择会影响评分或违背老师要求。
```

禁止用“基本完成”“应该可以”“看起来没问题”作为停止理由。

只有在所有完成标准都通过后，才允许输出：

```text
<promise>COURSE_PROJECT_GOAL_COMPLETE</promise>
```

## 实用提醒

高分不是靠“功能名多”，而是靠老师能看见、能点开、能听懂、能相信。

每轮迭代都要回答三个问题：

1. 这个改动是否让完整项目更稳定？
2. 这个改动是否让老师更容易觉得系统有特色？
3. 这个改动是否让前端更高级、更像真实产品？
4. 这个改动是否能被截图、日志、测试或答辩话术证明？
5. 是否还存在明显能继续提高分数的优化点？

如果答案显示还有高收益优化点，就继续迭代。只有当继续优化的收益已经很低，或者需要用户选择方向，才停止。
