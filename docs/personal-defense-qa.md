# 个人答辩 Q&A

## 1. 这个系统和普通聊天机器人有什么区别？

普通聊天机器人主要是把用户问题发给模型，然后显示回复。本系统先做业务处理，再做 AI 增强。它会识别售后意图、读取订单状态、检索知识库、判断是否已有售后申请、必要时创建人工工单，并把 AI 调用、检索和处理轨迹记录下来。因此它更像一个真实售后客服系统，而不是套壳聊天页面。

## 2. 为什么选择 Spring Boot + Vue 3 + MySQL + LangChain4j？

Spring Boot 适合实现稳定的后端业务服务，Vue 3 适合快速构建前端工作台，MySQL 能结构化保存订单、售后、会话、知识库和日志。LangChain4j 作为 Java 生态里的 AI 编排层，可以放在 Spring Boot 业务层之后做增强，不需要改变主业务架构。

## 3. LangChain4j 在系统中具体做了什么？

LangChain4j 用于调用 OpenAI-compatible 模型，并把用户问题、订单上下文、知识库命中、本地规则判断和业务工具结果组合成 prompt，让模型生成更自然的客服回复。它不是系统的业务规则引擎，不能绕过后端直接决定退换货结论。

## 4. AI 不可用时系统还能运行吗？

可以。系统保留本地规则兜底。`AiServiceImpl` 会判断 AI 是否启用、是否配置 API Key、模型调用是否成功。如果未启用、未配置或调用失败，聊天接口仍返回本地规则回复，并在 AI 日志中记录 `SKIPPED` 或 `FAILED`。这能保证答辩现场网络不稳定时核心功能仍可演示。

## 5. 怎么证明回复有知识库依据？

聊天时后端会按用户问题和意图检索 `knowledge_doc`，命中的文档会展示在咨询工作台右侧“知识命中”区域。同时检索结果会写入 `retrieval_log`，日志中心可以看到查询词、命中文档、排序、分数和命中原因。因此可以从页面和数据库日志两方面证明回复依据。

## 6. 多轮追问是怎么实现的？

系统会读取当前会话的最近消息、会话摘要和上一轮意图。如果当前问题包含“那、这个、刚才、怎么办、多久到账”等追问信号，或者问题较短且没有明确独立意图，就会结合上一轮意图进行上下文继承。例如上一轮是退货申请，本轮问“那多久到账”，系统会识别为退款进度。

## 7. 为什么不完全依赖大模型判断意图？

售后客服涉及订单状态、退换货规则和退款流程，不能把关键判断完全交给模型。模型可能产生幻觉或在网络失败时不可用。系统采用规则和业务数据优先、AI 增强靠后的设计，使核心业务稳定可控，同时保留 AI 带来的自然表达能力。

## 8. 人工工单什么时候会创建？

当用户表达投诉、人工客服、平台介入、商家一直不处理等诉求，或物流异常达到升级条件时，系统会调用工单服务创建或复用工单。工单包含用户问题、订单、会话、意图、优先级、AI 摘要和处理建议。

## 9. 工单为什么体现系统复杂度？

因为它把问答结果转化为后续业务处理对象。普通问答系统只回答用户问题，而本系统能把复杂或异常问题升级到人工客服，并保留状态流转、处理建议和日志证据，形成售后服务闭环。

## 10. 权限控制怎么实现？

后端使用 token 保存登录会话，`AuthInterceptor` 会拦截带 `@OperatorAnno` 的接口，要求用户角色为 `ADMIN`。前端路由通过 `adminOnly` 控制页面访问和菜单显示。客户角色只能访问咨询工作台和自己的订单售后入口，不能进入展示中心、知识库、订单管理、工单、日志或 AI 测试等后台页面。

## 11. 数据库为什么设计这么多表？

因为系统需要追踪完整业务链路。用户、订单、售后记录、知识库、会话、消息、意图、检索日志、AI 调用日志、处理轨迹和工单都承担不同职责。分表后能保证数据结构清晰，便于查询、展示和答辩解释。

## 12. 处理轨迹有什么用？

处理轨迹记录一次聊天从上下文解析到最终回复的每个步骤。它可以帮助开发调试，也能在答辩时解释系统为什么这样回答。老师如果质疑 AI 过程是否可解释，可以打开日志诊断中心，先看会话轨迹步骤摘要，再查看每一步原始轨迹。

## 13. 系统有哪些 RESTful 设计体现？

接口使用资源名复数和标准 HTTP 方法。例如 `GET /orders` 查询订单列表，`POST /chat-sessions` 创建会话，`POST /chat-sessions/{id}/messages` 发送消息，`GET /chat-sessions/{id}/process-traces` 查询会话处理轨迹，`GET /service-tickets` 查询工单。

## 14. 前端为什么新增答辩展示中心？

答辩时老师需要快速理解系统完整度和亮点。如果直接从某个后台页面开始，亮点比较分散。答辩展示中心把演示顺序、系统状态、8 个已实现亮点和关键页面入口集中起来，可以先给老师建立整体印象，再进入具体页面验证。

## 15. 你怎么保证没有伪造功能？

所有材料都对应真实代码、接口、数据库表和测试脚本。例如 AI 流式客服对应 `/chat-sessions/{id}/message-stream`，知识依据对应 `/knowledge-docs/search` 和 `retrieval_log`，工单对应 `/service-tickets` 和 `service_ticket`，权限对应 `AuthInterceptor` 和浏览器角色测试。

## 16. 最近一次验证做了哪些？

最近一次验证包括后端打包、前端构建、全链路接口烟测、浏览器主流程烟测和浏览器角色权限烟测。全链路接口烟测结果为 `FAILED_COUNT=0`，浏览器两项测试结果也都是 `FAILED_COUNT=0`。

## 17. 如果现场 sub2api 或模型服务失败怎么办？

先在系统总览或 AI 测试页确认 AI 状态。如果模型不可用，系统会显示本地兜底或失败状态，但咨询工作台仍可回答核心售后问题。答辩时可以主动说明这是系统稳定性设计：AI 只是增强层，不是唯一依赖。

## 18. 项目目前还有哪些不足？

当前知识库检索主要依赖 MySQL 文本检索和关键词匹配，语义相似问题还有提升空间；权限体系是轻量实现，还没有接入 Spring Security；日志诊断中心已经能展示 AI 成功率、平均耗时、知识命中和轨迹摘要，后续还可以扩展按日期聚合的趋势分析；前端部分后台页面还可以继续打磨产品质感。

## 19. 后续如何扩展？

可以扩展向量检索、细粒度 RBAC、售后状态工作流、意图统计看板、AI 成功率分析、客服处理时长分析、多渠道接入和更标准的 LangChain4j agent 工具执行链。

## 20. 你认为项目最有特色的地方是什么？

最有特色的是它把 AI 能力放进真实业务闭环，而不是停留在问答演示。系统能结合订单、知识库、多轮上下文和工单流程回答问题，还能在日志诊断中心用成功率、知识命中和轨迹步骤追踪处理过程，并且在 AI 不可用时保持本地兜底。
