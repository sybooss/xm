# 项目融合报告

## 1. 融合目标

本次融合以本项目当前版本为主干，保留 Spring Boot + Vue 3 + MySQL + LangChain4j 的整体结构，以及已有的登录、RBAC、SSE 流式聊天、LangChain4j 业务工具、人工工单和知识库能力。对同学项目只吸收能增强演示和业务闭环的功能，不直接覆盖认证、AI 配置和接口调用方式。

## 2. 已融合功能

### 2.1 客户侧订单与售后入口

- 在聊天工作台左侧补充用户信息区，提供“我的订单”和“退出”入口。
- 新增客户订单面板，可按当前登录用户筛选订单、搜索订单、选择订单并自动绑定到聊天订单号。
- 在客户订单面板中提供“申请售后”弹窗，支持退货退款、换货、仅退款三类申请。
- 后端新增 `POST /orders/{id}/after-sale-records`，通过当前登录用户校验订单归属，避免客户为他人订单提交售后。

### 2.2 会话体验增强

- 聊天会话列表增加删除按钮。
- 删除当前会话后自动清理当前聊天状态，并尽量切换到下一条会话。
- 聊天输入区保留 AI/本地兜底开关，并补充局部模型选择器，和系统状态里的模型切换能力保持一致。

### 2.3 订单后台管理增强

- 订单管理页由只读查询增强为后台维护页面。
- 管理员可新增、编辑、删除订单。
- 订单详情中可新增、编辑、删除售后记录。
- 售后记录变化后，后端会同步订单的 `after_sale_status`，删除售后记录后也会重新计算订单售后状态。

### 2.4 AI 回复规则增强

- 聊天订单上下文中加入该订单的售后记录和最新售后记录。
- 本地兜底回复会识别“订单已经有售后申请”的场景，避免继续引导用户重复提交退货或换货申请。
- AI 提示词中加入同样约束：如果订单已有售后申请，应说明当前状态和下一步，不要重复引导提交。

## 3. 没有直接合入的内容

- 未合入同学项目中的硬编码 AI Key。AI Key 必须继续通过环境变量或本机 `.env` 注入，不能写入源码或文档。
- 未合入同学项目中的简化认证实现。本项目继续使用现有的登录会话和 `@OperatorAnno` 管理员操作保护。
- 未合入前端直连 `http://localhost:8081` 的写法。本项目继续使用 `VITE_API_BASE_URL` 和 Vite 代理，方便本地和部署环境切换。
- 未替换已有 SSE 流式聊天、LangChain4j 业务工具和人工工单链路，这些是本项目当前更完整的能力。

## 4. AI 未启用的原因说明

如果页面显示 AI 未启用，通常不是融合代码缺失，而是运行环境没有打开 AI 或没有注入可用密钥。本项目应通过环境变量控制，例如：

- `AI_ENABLED=true`
- `OPENAI_API_KEY=本机密钥`
- `OPENAI_BASE_URL=本机 Sub2API 或 OpenAI 兼容地址`
- `OPENAI_MODEL=模型名`

本次融合不把任何真实 Key 写入仓库。没有上述运行配置时，系统会继续走本地兜底规则，保证演示和业务流程可用。

## 5. 融合后的结构取舍

本次融合采用“客户前台 + 管理后台”分工：

- 客户在聊天工作台查看自己的订单并提交售后申请。
- 管理员在订单管理页维护订单和售后记录。
- AI 只负责增强回复和辅助判断，不直接绕过后端业务校验。
- 售后状态以服务层统一同步，避免前端直接改状态造成数据不一致。

## 6. 验证结果

融合完成后已按项目规则执行：

- 后端编译：`mvn -q -DskipTests package`，通过。
- 前端构建：`npm run build`，通过，仅有 Vite 大 chunk 提醒。
- 全链路冒烟：`tools/full-smoke-test.ps1`，通过。当前默认 `AI_ENABLED=true`，配置真实模型密钥时验证 AI 增强链路；未配置密钥或模型异常时仍验证本地兜底链路。
- 浏览器冒烟：`npm run test:browser`，通过，覆盖登录、聊天、工单、知识库、订单、日志等页面。
- 客户专项冒烟：验证客户只能查看自己的订单、不能查询他人订单、不能重复提交进行中的售后申请、聊天回复能提示“无需重复提交”。
- 工作区检查：只包含本次融合相关源码、测试脚本和报告文件。

- 若需要 GitHub 的 `main` 也回到融合后的版本，应先确认是否覆盖远端同学融合提交，再决定是否合并分支或强推。

- 若要真实启用 AI，建议只在本机 `.env` 或部署环境变量中配置密钥，并保留本地兜底路径。
