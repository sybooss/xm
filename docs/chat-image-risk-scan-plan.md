# 聊天图片 AI 生成风险预审计划

## 1. 背景

当前系统已经完成两件事：

- 顾客在售后详情中上传图片凭证后，可以进入“凭证真实性审核”，展示 AI 生成风险、篡改风险、补证建议。
- 用户在在线咨询中可以发送图片，客服工单的原始会话也能看到图片。

但这两条链路之间仍有断点：用户如果直接在聊天中发送“坏产品照片”，系统目前只把它当作普通图片消息保存，右侧处理洞察不会提示这张图是否存在疑似 AI 生成、二次编辑、来源不清或证据不足风险。老师追问“聊天里发图后客服如何判断图片可信度”时，现有实现还不够完整。

## 2. 目标

第一版要实现“聊天图片风险预审”，让聊天图片进入可解释的客服判断链路。

1. 用户在 `/chat` 发送图片消息后，后端立即生成图片风险预审结果。
2. 预审结果返回给聊天页，右侧“处理洞察”显示：
   - 图片预审状态
   - 材料充分性
   - 真实性风险
   - AI 生成风险
   - 篡改风险
   - 元数据、水印/来源、视觉匹配信号
   - 建议补充材料
3. 预审结果写入 `process_trace`，作为本轮客服处理轨迹的一部分。
4. 工单原始会话中如果消息带图片，也能显示该图片对应的风险预审摘要。
5. AI 回复提示用户补充原始实拍图、故障视频、序列号、包装配件等材料，而不是直接按疑似 AI 图自动退款或驳回。

## 3. 非目标

第一版不做绝对鉴伪：

- 不声明“系统可以百分百判断图片是不是 AI 生成”。
- 不直接读取真实 EXIF、C2PA 或隐藏水印。
- 不因为图片风险高就自动退款、自动拒绝或自动关闭售后。
- 不新增复杂图片识别模型服务，避免引入额外部署风险。

页面和答辩话术统一使用“疑似 AI 生成风险”“来源/水印信号”“建议人工复核”，保持真实世界系统的可信边界。

## 4. 后端设计

### 4.1 风险对象

新增 `ChatImageRisk`，字段尽量对齐 `EvidenceAudit`：

- `auditStatus`：`PASS`、`NEED_MORE`、`RISKY`、`MANUAL_REVIEW`
- `sufficiencyLevel`：`SUFFICIENT`、`PARTIAL`、`INSUFFICIENT`
- `authenticityRisk`：`LOW`、`MEDIUM`、`HIGH`
- `aiGeneratedRisk`：`LOW`、`MEDIUM`、`HIGH`
- `tamperRisk`：`LOW`、`MEDIUM`、`HIGH`
- `metadataSignal`
- `visualSignal`
- `watermarkSignal`
- `requiredEvidence`
- `requiredEvidenceList`
- `summary`

### 4.2 扫描规则

输入来源：

- 用户消息正文
- 图片 URL
- 原始文件名
- contentType
- fileSize
- 订单商品名和售后状态

规则信号：

- 出现 `AI 生成`、`AIGC`、`生成图`、`midjourney`、`stable diffusion`、`dall-e`、`synthetic` 等词，提高 AI 生成风险。
- 出现 `无原图`、`网图`、`示意图`、`合成`、`非实拍`，提高来源不清风险。
- 出现 `Photoshop`、`P图`、`裁剪`、`拼接`、`马赛克`、`二次处理`，提高篡改风险。
- 出现 `EXIF`、`C2PA`、`隐形水印`、`hidden watermark`，记录为元数据/水印信号，建议人工查看原始文件。
- 如果图片说明过短或无法对应商品问题，要求补充故障视频、序列号照片、包装配件照片或物流截图。

### 4.3 处理轨迹

新增流程节点：

```text
CHAT_IMAGE_RISK_SCAN
```

当本轮无图片时记录 `SKIPPED`；有图片时记录 `SUCCESS`，`detail_json` 保存 `ChatImageRisk` 的完整结果。

### 4.4 AI 回复上下文

`ChatServiceImpl` 生成 prompt 时加入“聊天图片风险预审”片段，让 AI 回复更像真实客服：

- 风险低：提示继续按正常售后规则处理。
- 风险中高：提示用户补充原始实拍、故障视频、序列号或物流截图。
- 不直接承诺退款或拒绝。

## 5. 前端设计

### 5.1 聊天页

`ChatWorkbenchView.vue` 在右侧“处理洞察”增加“图片真实性预审”卡片。

显示内容：

- 顶部摘要
- `StatusTag` 展示状态和风险等级
- 补证建议
- 元数据、水印/来源、视觉信号

消息气泡中如果该图片消息带 `imageRisk`，在图片下方显示一个小型风险摘要。

### 5.2 工单页

`ServiceTicketView.vue` 的“原始会话”中，如果图片消息带 `imageRisk`，显示同样的小型风险摘要。这样人工客服接管时不需要回到聊天页，也能看到用户发图的风险预审。

## 6. 数据库策略

第一版不新增表，复用 `process_trace` 存储聊天图片预审结果，原因：

- 聊天图片预审属于本轮会话处理轨迹，不一定已经进入正式售后申请。
- 售后申请阶段已有 `evidence_audit` 表，不能把聊天预审和正式凭证审核混在一起。
- `process_trace` 已经用于解释聊天的上下文、意图、知识命中、AI 生成和工单判断，放入图片预审符合现有框架。

后续如果要做更强的图片取证，可以再把聊天图片转为正式 `after_sale_evidence` 并生成 `evidence_audit`。

## 7. 验证计划

需要补充并执行：

1. `mvn -q -DskipTests package`
2. `npm run build`
3. `tools/full-smoke-test.ps1`
   - 断言图片聊天消息返回 `imageRisk`
   - 断言 `process_trace` 包含 `CHAT_IMAGE_RISK_SCAN`
   - 断言工单原始会话可拿到图片风险预审
4. `npm run test:browser`
   - 页面上传聊天图片
   - 看到“图片真实性预审”
   - 工单原始会话看到聊天图片和风险摘要

## 8. 自审结论

这个方案补上了“聊天发图”和“AI 图片风险识别”之间的断层，同时没有夸大成绝对鉴伪。它符合现有 Spring Boot + Vue 3 + MySQL + LangChain4j 架构：

- 后端仍由业务规则先做可靠判断，AI 只作为回复增强。
- 轨迹进入 `process_trace`，便于日志和答辩解释。
- 前端只增加风险卡片，不改变主流程。
- 不新增大表和外部模型服务，风险可控。

需要注意的瑕疵：第一版不能真正读取隐藏水印或 EXIF，只能把“隐藏水印/来源信息需要人工复核”作为信号和建议。答辩时应表述为“风险预审 + 人工复核入口”，不要表述为“系统已经百分百识别 AI 图片”。
