# 凭证图片上传与 AI 生成风险审核补强计划

## 1. 问题定位

原“AI 凭证真实性审核”已经实现 `evidence_audit`、风险信号、补证建议和顾客/管理员展示，但缺少真实图片上传入口。结果是用户只能填写文字或图片链接，无法把“坏产品图片”真正提交给客服，答辩时会削弱“识别疑似 AI 生成图片凭证”的可信度。

本补强目标是把图片入口打通：

1. 顾客补充凭证时可以选择本地图片上传。
2. 后端保存图片到运行目录，返回可访问 URL。
3. `after_sale_evidence` 保存 `evidence_type=IMAGE`、`file_url` 和说明。
4. 顾客端和管理员端都能预览图片。
5. 凭证审核继续基于图片 URL、说明文本和售后上下文识别疑似 AI 生成、水印、EXIF、编辑软件、二次处理等风险信号。

## 2. 实现边界

- 第一版不承诺真实读取隐藏水印、C2PA 或完整 EXIF，只把文件名、URL、用户说明和上下文作为风险信号。
- 不自动退款、不自动驳回，只写凭证、审核记录和处理日志。
- 上传文件只允许常见图片格式，限制大小，避免把该功能做成通用文件网盘。
- 运行文件放入 `tmp/uploads/evidences`，不提交到 Git。

## 3. 后端改动

新增：

- `EvidenceFileService`
- `EvidenceFileServiceImpl`
- `EvidenceUploadController`

接口：

```http
POST /customer/evidence-files
```

请求为 `multipart/form-data`，字段名 `file`。返回：

```json
{
  "fileUrl": "/uploads/evidences/20260513/xxx.png",
  "originalFilename": "earphone-ai-generated.png",
  "size": 12345
}
```

静态资源：

- Spring Boot 暴露 `/uploads/**` 到本地 `tmp/uploads/`。

## 4. 前端改动

顾客售后详情的补充凭证弹窗：

- 增加图片上传控件。
- 图片上传成功后自动切换 `evidenceType=IMAGE`。
- 表单提交时把 `fileUrl` 一起提交。
- 如果凭证类型是图片，展示预览图和“疑似 AI 生成/水印/编辑痕迹说明”输入提示。

顾客和管理员凭证列表：

- 若 `fileUrl` 存在，显示图片预览。
- 继续展示审核面板中的 AI 生成风险、篡改风险和补证建议。

## 5. 自审修订

初稿风险：

1. 如果只上传不预览，老师仍然看不出图片真的进入系统。
2. 如果文案说“识别隐藏水印”，但代码没有解析，会显得夸张。
3. 如果上传接口不做类型和大小限制，安全边界不好。

修订结果：

采用“真实上传 + 可预览 + 风险信号审核”的路线。文案只说“疑似 AI 生成图片凭证风险识别”，不说绝对鉴定。这样既能满足老师说的亮点，也符合真实售后系统的可信边界。

## 6. 验证标准

- 后端编译通过。
- 前端构建通过。
- 顾客端可上传图片并提交为 `IMAGE` 凭证。
- 管理员审核页可看到图片预览和凭证审核结果。
- 全链路冒烟和浏览器冒烟通过。
