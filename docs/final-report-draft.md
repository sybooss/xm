# 《复杂软件系统实践》结项报告草稿

## 一、系统简介

本项目为“电商退换货智能客服系统”，面向电商售后中高频出现的退货申请、换货申请、退款进度、物流异常、规则咨询、投诉与人工转接等场景。系统采用 `Spring Boot + Vue 3 + MySQL + LangChain4j` 技术栈，目标不是只实现一个聊天框，而是构建一条可演示、可追踪、可维护的售后客服业务闭环。

系统的核心流程为：

```text
用户登录/注册
-> 绑定订单并发起售后咨询
-> 后端识别售后意图
-> 读取订单和售后上下文
-> 检索知识库规则
-> 生成本地业务判断
-> 调用 LangChain4j 进行 AI 增强
-> 失败时走本地规则兜底
-> 必要时创建人工客服工单
-> 保存消息、意图、检索、AI 调用和处理轨迹日志
```

项目当前已实现完整的前后端分离系统。后端负责 RESTful 接口、业务编排、MySQL 数据访问、知识检索、AI 调用、日志追踪和权限控制；前端负责登录注册、答辩展示中心、咨询工作台、知识库管理、订单管理、人工工单、日志中心和 AI 测试页面。数据库通过结构化表保存用户、订单、售后记录、会话消息、知识文档、意图识别、检索日志、AI 调用日志、处理轨迹和人工工单。

## 二、主要功能

### 2.1 登录注册与权限控制

系统提供管理员和客户两类角色。登录注册页面支持管理员演示账号 `admin / 123456` 和客户账号 `demo_customer / 123456`，也支持客户自助注册。注册客户默认角色为 `CUSTOMER`，只能访问咨询工作台和自己的订单售后入口；管理员可访问系统总览、答辩展示中心、知识库、订单管理、人工工单、日志中心和 AI 测试等后台功能。

权限控制由后端 `AuthInterceptor`、`AuthServiceImpl` 和 `@OperatorAnno` 注解配合完成。对新增、修改、删除和后台管理类接口，后端会校验 `Authorization` token 并确认用户角色为 `ADMIN`；前端路由也通过 `adminOnly` 元信息限制客户访问后台页面。浏览器角色测试已经覆盖客户无法访问 `/showcase`、`/dashboard`、`/knowledge`、`/orders`、`/service-tickets`、`/logs` 和 `/ai-test` 的场景。

### 2.2 售后咨询工作台

咨询工作台是系统的核心业务页面。用户可以创建会话、绑定订单号、输入售后问题，并在中间消息区看到 AI 或本地兜底回复。右侧“处理洞察”面板展示意图识别、上下文承接、订单上下文、知识命中、人工转接和回答过程。

当前演示订单以 `DD202604290001` 为主，系统内置多条测试订单覆盖可退货、退款中、物流异常等场景。用户输入“这个订单能不能退货？”后，后端会识别为 `RETURN_APPLY`，查询订单是否已签收、售后状态是否为空，并检索“七天无理由退货规则”等知识文档。继续追问“那退款一般多久到账？”时，系统会通过会话上下文判断这是上一轮退货问题后的追问，并将本轮意图解析为 `REFUND_PROGRESS`。

### 2.3 知识库和 RAG 依据展示

知识库分为退货规则、换货规则、退款规则、物流异常、投诉与人工转接和通用 FAQ 等分类。每个知识文档包含标题、文档类型、适用意图、问题、答案、正文、关键词、优先级和状态。

聊天链路中，后端会调用 `knowledgeDocMapper.search` 按用户问题和意图检索知识文档，并将命中文档写入 `retrieval_log`。前端右侧面板展示命中文档，日志中心展示查询词、命中文档、排序、分数和命中原因。这样老师在答辩时能够看到 AI 回复不是凭空生成，而是结合了真实业务规则。

### 2.4 LangChain4j AI 增强与本地兜底

后端通过 `AiServiceImpl` 接入 LangChain4j 的 OpenAI-compatible 模型调用能力，支持本机 `sub2api` 或其他兼容网关。`ChatServiceImpl` 会把用户问题、会话摘要、意图识别、订单上下文、知识命中、本地业务判断和业务工具结果组织成 prompt，再交给模型生成更自然的客服回复。

AI 只作为增强层，不替代业务规则。订单是否存在、是否可退换、是否已有售后申请、是否需要人工工单，都由 Spring Boot 业务服务判断。模型调用失败、未配置 key 或主动关闭 AI 时，系统会返回本地规则模板回复，并在 `ai_call_log` 中记录 `FAILED` 或 `SKIPPED` 状态，保证答辩现场即使网络或模型不可用也能稳定演示。

### 2.5 LangChain4j 业务工具调用

系统将订单查询、知识库检索和工单创建封装到 `AiBusinessToolServiceImpl`，并使用 LangChain4j `@Tool` 风格标注。这些工具包括：

- `queryOrderStatus`：根据订单号查询订单、物流和售后状态。
- `searchAfterSaleKnowledge`：按问题和意图检索知识库依据。
- `createServiceTicket`：投诉、商家不处理或物流异常时创建人工客服工单。

当前实现中，业务工具结果会被注入模型上下文，并在处理轨迹中记录 `BUSINESS_TOOL_CALLS` 步骤。这体现了 AI 与业务系统的结合，而不是简单调用一个通用聊天接口。

### 2.6 智能工单升级

当用户表达“投诉”“人工客服”“商家一直不处理”“平台介入”等诉求，或订单物流状态异常并触发人工处理条件时，系统会创建或复用人工客服工单。工单保存工单号、会话、消息、订单、用户、意图、优先级、状态、客户问题、AI 摘要、处理建议和处理人。

前端“人工工单”页面支持按关键词、状态和优先级筛选工单，并展示 AI 摘要和处理建议。客服可以把工单状态从 `PENDING` 修改为 `PROCESSING`、`RESOLVED` 或 `CLOSED`。该功能使系统从问答工具升级为包含人工协同的售后服务系统。

### 2.7 日志追踪与可解释性

系统保存三类关键日志：

- `ai_call_log`：记录提供方、模型、请求摘要、响应摘要、状态、耗时和错误。
- `retrieval_log`：记录查询词、命中文档、排序、分数、命中原因和快照。
- `process_trace`：记录上下文解析、意图识别、订单上下文、知识检索、业务工具调用、AI 生成、工单判定和最终回复。

日志诊断中心在保留 AI 调用日志、知识检索日志和处理轨迹三个原始证据标签页的基础上，新增诊断总览：聚合展示 AI 成功率、平均耗时、知识命中数量、去重命中文档、平均检索分数、会话轨迹步骤和高频命中文档。答辩时可以在演示完聊天后进入该页面，先用统计指标说明系统稳定性和可解释性，再切换到原始日志证明每次回复背后都有可追踪的处理链路。

### 2.8 答辩展示中心

为提高现场展示效率，系统新增 `/showcase` 答辩展示中心。管理员登录后默认进入该页面。页面以克制、精致的产品化风格呈现系统主题、数据库状态、AI 状态、当前模型、兜底策略、演示流程、8 个已实现亮点、答辩讲解骨架和现场检查清单。

该页面不是静态宣传页，而是系统真实能力的索引。右侧提供咨询工作台、知识库、人工工单、日志中心和 AI 测试入口，便于老师按一条主线检查完整项目。

## 三、关键实现说明

### 3.1 后端分层设计

后端采用 Controller、Service、Mapper、Pojo 的分层结构：

- Controller 负责接收参数、权限入口和统一返回。
- Service 负责业务编排、事务边界、AI 调用和本地兜底。
- Mapper 负责 MyBatis 数据访问，SQL 写在 `resources/mapper/*.xml` 中。
- Pojo 保存实体、请求对象、分页对象和统一响应。

接口采用资源名词和标准 HTTP 方法，例如 `GET /service-tickets`、`POST /chat-sessions/{id}/messages`、`GET /orders/{id}/after-sale-records`。这种结构便于维护，也符合课程对程序设计规范的要求。

### 3.2 聊天主链路实现

聊天主链路集中在 `ChatServiceImpl.sendMessage`。该方法在一个事务内完成消息保存、上下文解析、意图识别、订单查询、知识库检索、本地回复、AI 增强、工单判定、日志记录和最终响应组装。

处理轨迹中的关键步骤包括：

```text
CONTEXT_RESOLVE
INTENT_RECOGNIZE
ORDER_CONTEXT
KNOWLEDGE_RETRIEVAL
BUSINESS_TOOL_CALLS
AI_GENERATION
HUMAN_TICKET_CHECK
TICKET_CREATED
FINAL_REPLY
```

这些步骤既能帮助开发调试，也能在答辩时解释“系统为什么这样回答”。

### 3.3 多轮上下文实现

多轮上下文通过 `chat_session.summary`、最近消息和上轮意图实现。后端会判断当前问题是否包含“那、这个、刚才、怎么办、多久到账”等追问信号，或者问题较短且缺少独立意图词。如果满足条件，就结合上轮意图和最近消息进行意图继承。例如上一轮是退货申请，本轮追问“多久到账”，系统会解析为退款进度。

该方案的好处是稳定、可解释，不完全依赖大模型；即使 AI 不可用，本地规则也能完成常见追问。

### 3.4 前端产品化设计

前端采用 Vue 3、Vite、Pinia、Vue Router、Axios 和 Element Plus。页面结构包括固定侧栏、顶部状态栏和内容区。核心页面按业务角色组织：

- `/showcase`：答辩展示中心。
- `/chat`：咨询工作台。
- `/knowledge`：知识库管理和检索调试。
- `/orders`：订单和售后上下文。
- `/service-tickets`：人工工单。
- `/logs`：AI、检索和处理轨迹日志。
- `/ai-test`：模型连通性测试。

为了避免普通后台模板感，答辩展示中心使用白底、轻边框、留白、清晰层级和稳定尺寸，将复杂能力整理成演示路径和亮点卡片。咨询工作台则保持三栏布局，适合边演示边解释系统链路。

## 五、项目特色与创新点

1. 系统不是单一聊天框，而是完整售后客服业务闭环。
2. RAG 知识库命中结果可在页面和日志中追溯。
3. LangChain4j 工具调用把订单查询、知识检索和工单能力接入 AI 上下文。
4. 多轮追问既能通过上下文继承理解省略问题，也能在本地规则下稳定运行。
5. AI 失败不影响核心业务，系统保留本地规则兜底，适合现场答辩。
6. 投诉和异常物流可以自动升级人工工单，体现真实客服协同。
7. 日志诊断中心提供 AI 调用、检索和处理轨迹，并聚合 AI 成功率、平均耗时、知识命中和会话步骤，便于证明系统可解释、可调试。
8. 答辩展示中心把完整项目能力整理成可讲、可点、可验证的高分入口。

## 六、团队分工建议

如果按团队项目汇报，可将分工组织为：

| 成员方向 | 主要工作 |
| --- | --- |
| 后端业务与数据库 | 设计 MySQL 表结构，实现订单、售后、知识库、会话、工单和日志接口 |
| AI 与客服链路 | 实现意图识别、多轮上下文、RAG 检索、LangChain4j 接入和本地兜底 |
| 前端页面与交互 | 实现登录注册、咨询工作台、展示中心、知识库、工单、日志和 AI 测试页面 |
| 测试与文档 | 编写 README、接口文档、数据库文档、演示脚本、测试用例和烟测脚本 |

个人答辩时，应从完整项目中提炼自己的负责模块，而不是把系统拆成互不相关的小块。

## 七、存在不足与未来拓展

当前系统已能支撑课程结项答辩，但仍有进一步优化空间：

1. 知识库检索目前主要基于 MySQL 文本检索和关键词匹配，后续可加入向量检索，提高语义相似问题的命中率。
2. 登录权限目前是轻量自实现，后续可接入 Spring Security、刷新 token 和更细粒度 RBAC。
3. 日志诊断中心已能展示 AI 成功率、平均耗时、知识命中和处理轨迹摘要，后续可继续扩展意图分布、工单趋势、客服处理时长和按日期聚合的趋势看板。
4. 当前演示订单是固定种子数据，后续可接入真实电商订单系统或模拟更复杂的售后状态流转。
5. 现有 AI 工具调用主要将业务工具结果注入 prompt，后续可进一步扩展为更标准的 agent 工具执行链。
6. 前端已经具备答辩展示中心，但知识库、日志和工单页面还可以继续提升产品质感和数据可视化表现。

这些拓展方向都建立在当前可运行系统之上，具有可行性，也能作为结项报告中的合理后续建议。
