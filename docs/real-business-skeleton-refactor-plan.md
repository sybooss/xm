# 真实业务骨架改造方案

## 1. 改造目标

当前系统已经具备聊天、订单、知识库、工单、日志和 AI 辅助能力，但整体仍偏向“AI 客服演示器”。它能展示 AI 回复和一些运营看板，却还不像现实中可以试用的售后客服系统。

本轮改造目标不是继续增加展示型特色功能，而是把项目升级为一个有真实业务骨架的售后客服系统：

- 顾客端不再只围绕聊天，而是围绕订单、售后申请、凭证、进度、补充材料和评价。
- 管理员端不再只看面板，而是围绕售后审核、工单分派、处理记录、SLA 和客户历史。
- AI 不再作为主业务入口，而是作为辅助层，负责摘要、风险识别、知识推荐、回复草稿和处理建议。
- 后端必须有明确状态机、处理记录、权限边界和可审计证据。
- 前端必须明显区分顾客工作台和管理员工作台，让系统看起来像真实业务系统，而不是聊天 demo。

## 2. 当前系统的核心问题

### 2.1 顾客端业务太薄

现在顾客主要是在咨询工作台发消息，体验上更像“问 AI 一个售后问题”。真实售后系统里，顾客最常见的行为不是聊天，而是：

- 查看自己的订单。
- 选择订单申请退货、换货、退款或投诉。
- 填写原因和说明。
- 上传图片、物流单号、商品凭证。
- 查看售后处理进度。
- 按要求补充材料。
- 确认问题已解决。
- 对客服处理进行评价。

这些流程当前没有形成完整闭环。

### 2.2 管理员端区分度不够

管理员现在能看订单、工单、知识库、日志和运营页面，但缺少真实处理动作。现实管理员需要的是任务队列和处理台：

- 待审核售后申请队列。
- 待补充材料队列。
- 待客户寄回队列。
- 待仓库收货队列。
- 待退款队列。
- 投诉升级队列。
- SLA 即将超时队列。
- 每个售后单的处理备注、状态流转、负责人、操作记录。

当前管理员更多是在“看数据”，不是在“处理业务”。

### 2.3 AI 占比过高

当前系统容易给人一种感觉：核心能力就是 AI 回复。真实项目中，AI 应该是辅助层，而不是业务规则的替代品。

真实落地时应该遵守：

- AI 可以建议，不直接决定退款。
- AI 可以总结，不直接修改订单状态。
- AI 可以推荐知识，不直接绕过人工审核。
- AI 可以生成回复草稿，但管理员应能确认、修改或驳回。
- AI 失败时，售后流程仍然可以完整运行。

### 2.4 缺少严格状态机

售后业务不能只靠几个状态字段展示。它需要明确规定：

- 当前状态能执行哪些动作。
- 哪些角色可以执行动作。
- 执行动作需要哪些字段。
- 状态能流向哪里。
- 哪些状态不可回退。
- 每次流转是否必须写处理记录。

没有状态机，系统就更像样例 CRUD，而不是业务系统。

## 3. 目标产品形态

改造后系统应拆成三个清晰入口。

### 3.1 顾客端：我的售后中心

顾客登录后默认进入“我的售后中心”，而不是管理员展示页。

顾客端核心页面：

| 页面 | 路径建议 | 作用 |
| --- | --- | --- |
| 我的订单 | `/customer/orders` | 查看自己的订单，选择申请售后 |
| 申请售后 | `/customer/after-sales/apply` | 按订单发起退货、换货、退款、投诉 |
| 我的售后 | `/customer/after-sales` | 查看售后申请列表和状态 |
| 售后详情 | `/customer/after-sales/{id}` | 查看进度时间线、处理记录、补充材料入口 |
| 在线咨询 | `/customer/chat` | AI/客服辅助入口，不作为唯一主入口 |
| 我的评价 | `/customer/reviews` | 对已完成售后处理进行评价 |

顾客端要体现“我提交了什么、现在到哪一步、下一步要我做什么”。

### 3.2 管理员端：售后处理工作台

管理员登录后默认进入“售后处理工作台”。

管理员端核心页面：

| 页面 | 路径建议 | 作用 |
| --- | --- | --- |
| 审核队列 | `/admin/after-sales/review` | 审核顾客提交的售后申请 |
| 工单队列 | `/admin/tickets` | 处理投诉、异常、人工转接 |
| 售后详情处理台 | `/admin/after-sales/{id}` | 修改状态、写备注、查看证据链 |
| SLA 中心 | `/admin/sla` | 查看即将超时和已超时事项 |
| 客户画像 | `/admin/customers/{id}` | 查看客户历史订单、售后、投诉、评价 |
| 知识库运营 | `/admin/knowledge` | 维护规则、审核 AI 未命中问题 |
| AI 质检中心 | `/admin/ai-quality` | 查看 AI 摘要、回复草稿、风险识别质量 |

管理员端要体现“我今天有哪些任务、每个任务怎么处理、处理后留下什么证据”。

### 3.3 AI 辅助层：从主角降为副驾驶

AI 入口保留，但角色调整为辅助工具：

| AI 能力 | 真实业务落点 |
| --- | --- |
| 意图识别 | 帮助判断用户想退货、换货、退款、投诉 |
| 知识推荐 | 给客服推荐可引用的规则 |
| 对话摘要 | 自动生成工单摘要，减少客服录入 |
| 风险识别 | 标记高情绪、高金额、重复投诉、SLA 风险 |
| 回复草稿 | 生成客服回复草稿，由客服确认发送 |
| 质检评分 | 检查回复是否包含规则、进度、安抚和下一步动作 |

AI 不应该直接完成退款、驳回、改状态等关键动作。

## 4. 需要新增或强化的数据模型

### 4.1 售后申请主表 `after_sale_application`

用于替代当前偏简单的售后记录，成为真实流程主对象。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `application_no` | 售后申请编号 |
| `order_id` | 关联订单 |
| `user_id` | 顾客 |
| `service_type` | RETURN / EXCHANGE / REFUND / COMPLAINT |
| `reason_code` | 标准原因 |
| `reason_text` | 用户填写说明 |
| `status` | 状态机状态 |
| `refund_amount` | 申请退款金额 |
| `approved_amount` | 审核通过金额 |
| `priority` | LOW / NORMAL / HIGH / URGENT |
| `sla_deadline` | SLA 截止时间 |
| `assigned_to` | 当前处理人 |
| `ai_summary` | AI 摘要 |
| `risk_level` | 风险等级 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `closed_at` | 关闭时间 |

### 4.2 售后凭证表 `after_sale_evidence`

用于保存顾客上传的图片、视频、物流单号或文字凭证。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `application_id` | 售后申请 |
| `evidence_type` | IMAGE / VIDEO / TEXT / LOGISTICS_NO |
| `file_url` | 文件地址 |
| `content` | 文本内容 |
| `uploaded_by` | 上传人 |
| `created_at` | 创建时间 |

### 4.3 处理记录表 `after_sale_process_log`

每一次状态变化、备注、驳回、补充材料都要留痕。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `application_id` | 售后申请 |
| `operator_id` | 操作人 |
| `operator_role` | CUSTOMER / ADMIN / SYSTEM / AI |
| `action` | SUBMIT / APPROVE / REJECT / REQUEST_MORE_EVIDENCE / REFUND / CLOSE |
| `from_status` | 原状态 |
| `to_status` | 新状态 |
| `remark` | 处理备注 |
| `created_at` | 创建时间 |

### 4.4 客服回复草稿表 `reply_draft`

AI 生成的内容不直接发送，先形成草稿。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `application_id` | 售后申请 |
| `ticket_id` | 可选工单 |
| `draft_content` | AI 草稿 |
| `source_type` | AI / TEMPLATE / MANUAL |
| `status` | DRAFT / USED / DISCARDED |
| `created_by` | 创建人或系统 |
| `created_at` | 创建时间 |

### 4.5 客户评价表 `service_review`

用于闭环售后体验。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `application_id` | 售后申请 |
| `user_id` | 顾客 |
| `rating` | 1-5 分 |
| `tags` | 响应快、态度好、处理慢等 |
| `comment` | 评价内容 |
| `created_at` | 创建时间 |

## 5. 售后状态机设计

建议使用统一状态机，不再让状态随意修改。

### 5.1 通用状态

| 状态 | 含义 | 顾客可操作 | 管理员可操作 |
| --- | --- | --- | --- |
| `SUBMITTED` | 已提交 | 撤销、补充说明 | 审核 |
| `UNDER_REVIEW` | 审核中 | 查看进度 | 通过、驳回、要求补充 |
| `NEED_MORE_EVIDENCE` | 待补充凭证 | 上传凭证 | 查看凭证 |
| `APPROVED` | 已通过 | 查看下一步 | 指派处理 |
| `WAIT_BUYER_SEND` | 待买家寄回 | 填写物流单号 | 查看 |
| `WAIT_SELLER_RECEIVE` | 待商家收货 | 查看 | 确认收货 |
| `REFUNDING` | 退款中 | 查看 | 确认退款 |
| `EXCHANGING` | 换货中 | 查看 | 发出换货商品 |
| `REJECTED` | 已驳回 | 查看原因、重新申请 | 关闭 |
| `COMPLETED` | 已完成 | 确认、评价 | 归档 |
| `CANCELLED` | 已取消 | 无 | 归档 |

### 5.2 关键规则

- 顾客只能操作自己的售后申请。
- 管理员才能审核、驳回、退款、关闭。
- 任何状态变化都必须写入 `after_sale_process_log`。
- 驳回必须填写驳回原因。
- 要求补充凭证必须填写补充说明。
- 退款金额不能超过订单实付金额。
- 物流异常、重复投诉、高金额订单自动提升优先级。
- SLA 超时后自动产生风险标记，但不自动退款。

## 7. 前端页面改造

### 7.1 顾客端首页

顾客登录后不再进入通用聊天页，而是进入“我的售后中心”。

首屏应包含：

- 最近订单。
- 正在处理的售后申请。
- 需要我补充的事项。
- 在线咨询入口。
- 常见售后规则。

### 7.2 顾客售后申请向导

建议做成 4 步：

1. 选择订单。
2. 选择售后类型：退货 / 换货 / 退款 / 投诉。
3. 填写原因、金额、说明并上传凭证。
4. 确认提交，生成售后申请编号。

### 7.3 顾客售后详情页

必须展示：

- 当前状态。
- 下一步需要顾客做什么。
- 售后进度时间线。
- 管理员处理备注。
- 已上传凭证。
- 继续补充材料按钮。
- 在线咨询按钮。
- 完成后评价按钮。

### 7.4 管理员工作台

管理员首页应从“展示看板”变成“处理队列”。

首屏应包含：

- 待审核数量。
- 待补充材料数量。
- SLA 即将超时数量。
- 高优先级投诉数量。
- 今日已处理数量。
- 我的待办列表。

### 7.5 管理员售后处理台

处理台应包含：

- 顾客信息。
- 订单信息。
- 售后申请信息。
- 凭证材料。
- 对话记录。
- AI 摘要和风险提示。
- 处理记录时间线。
- 状态操作按钮。
- 回复草稿区。

## 8. AI 能力重新定位

### 8.1 保留

- 对话摘要。
- 意图识别。
- 知识库推荐。
- 风险识别。
- 回复草稿。
- 质检评分。

### 8.2 限制

- AI 不直接退款。
- AI 不直接驳回。
- AI 不直接修改售后状态。
- AI 不直接发送最终客服回复。
- AI 不绕过权限和状态机。

### 8.3 新增 AI 审计字段

每次 AI 建议都应记录：

- 输入摘要。
- 使用知识条目。
- 输出建议。
- 置信度。
- 是否被管理员采纳。
- 管理员修改后的最终内容。

这样 AI 才能从“会回复”变成“可审计的辅助决策系统”。

## 9. 权限边界

### 9.1 顾客权限

顾客只能：

- 查看自己的订单。
- 创建自己的售后申请。
- 查看自己的售后详情。
- 上传自己的凭证。
- 取消自己的未处理申请。
- 评价自己的已完成申请。
- 发起咨询。

### 9.2 管理员权限

管理员可以：

- 查看所有售后申请。
- 审核、驳回、要求补充。
- 分派工单。
- 修改售后状态。
- 处理退款和换货。
- 维护知识库。
- 查看 AI 质量和日志。

### 9.3 后续可扩展角色

如果要更像真实企业系统，可以继续拆分：

- `CUSTOMER`：顾客。
- `SERVICE_AGENT`：客服。
- `SERVICE_MANAGER`：客服主管。
- `WAREHOUSE_STAFF`：仓库人员。
- `FINANCE_STAFF`：财务退款人员。
- `ADMIN`：系统管理员。

## 10. 分阶段实施计划

### 阶段一：建立真实售后主流程

目标：系统从“聊天 demo”转为“售后申请系统”。

任务：

- 新增 `after_sale_application`。
- 新增 `after_sale_process_log`。
- 实现售后状态机。
- 新增顾客售后申请向导。
- 新增顾客售后详情页。
- 新增管理员审核队列。

验收：

- 顾客可以从订单发起售后申请。
- 管理员可以审核通过或驳回。
- 顾客可以看到状态时间线。
- 每次状态变化都有处理记录。

### 阶段二：补齐凭证、工单和 SLA

目标：让流程具有真实处理能力。

任务：

- 新增凭证上传。
- 新增处理备注。
- 新增 SLA 截止时间。
- 新增超时扫描。
- 工单和售后申请关联。
- 管理员工作台显示待办队列。

验收：

- 顾客能上传图片或物流单号。
- 管理员能要求补充材料。
- SLA 超时任务能被识别。
- 投诉类申请自动进入高优先级队列。

### 阶段三：AI 辅助改造

目标：AI 从聊天主角变成客服副驾驶。

任务：

- 售后详情页生成 AI 摘要。
- AI 推荐知识库规则。
- AI 生成回复草稿。
- 管理员确认后才能使用草稿。
- 记录 AI 建议是否被采纳。

验收：

- 没有 AI 时流程仍能跑完。
- AI 只提供建议，不直接修改关键状态。
- 管理员可修改 AI 回复草稿。
- AI 建议有审计记录。

### 阶段四：客户画像和运营闭环

目标：让系统具备真实客服管理价值。

任务：

- 客户历史订单。
- 客户历史售后。
- 客户历史投诉。
- 客户满意度评价。
- 重复投诉识别。
- 知识缺口沉淀。

验收：

- 管理员能看到客户历史。
- 重复投诉能被标记。
- 售后完成后顾客能评价。
- 评价和投诉数据能进入运营统计。

## 11. 优先级排序

如果时间有限，建议按下面顺序做：

| 优先级 | 改造项 | 原因 |
| --- | --- | --- |
| P0 | 售后申请主流程和状态机 | 决定系统是否像真实业务 |
| P0 | 顾客端我的订单和申请售后 | 拉开顾客端与管理员端差异 |
| P0 | 管理员审核队列和处理台 | 让管理员真正能处理业务 |
| P1 | 处理记录和凭证上传 | 形成可追踪证据 |
| P1 | SLA 和优先级队列 | 接近真实客服运营 |
| P1 | AI 摘要和回复草稿 | 保留 AI 亮点但不喧宾夺主 |
| P2 | 客户画像和评价 | 增强真实业务价值 |
| P2 | AI 质检和知识缺口 | 增强答辩亮点 |

## 12. 改造完成后的判断标准

当系统满足下面条件时，才可以说它不再是玩具：

- 顾客不需要聊天，也能完整提交和跟踪售后。
- 管理员每天登录后能看到明确待办任务。
- 售后申请有严格状态机。
- 每个处理动作都有记录。
- 顾客和管理员看到的是完全不同的工作台。
- AI 只是辅助建议，业务流程不依赖 AI 成功。
- 售后单、工单、订单、凭证、评价、日志能串成一条证据链。
- 数据库结构能支撑真实业务扩展。
- 测试覆盖顾客提交、管理员审核、补充凭证、完成评价等主流程。

## 13. 一句话总结

这个项目下一步最重要的不是“让 AI 更会说”，而是“让售后业务真的跑起来”。AI 应该站在客服和管理员旁边帮忙，而不是站在系统中央假装业务流程。

## 15. 每轮改造闭环

每个版本必须按下面顺序执行：

1. 开始前先读 `docs/real-business-skeleton-refactor-progress.md`，确认当前版本、遗留问题和本轮范围。
2. 只修改本轮相关文件，避免顺手改 PPT、论文、答辩材料、运行产物或无关历史文件。
3. 代码完成后执行与范围匹配的验证。后端至少跑 `mvn -q -DskipTests package`，前端至少跑 `npm run build`，涉及接口或页面时补充 `tools/full-smoke-test.ps1` 或 `npm run test:browser`。
4. 验证通过后重新审视项目，至少回答：本轮是否真的让系统更像真实双端业务系统、是否存在 AI 越权、是否存在状态机漏洞、是否有明显下一步高价值优化。
5. 把版本结果、验证命令、遗留问题和下一轮建议写入进度文档。
6. 运行 `git status --short`，确认只包含本轮相关文件。
7. 提交并推送到 `origin`。如直连失败，按 `AGENTS.md` 使用本机代理推送。

## 16. 停止优化标准

只有同时满足下面条件，才认为“从客服演示到真实双端系统”的改造可以停止：

- V1 到 V7 的主线版本均已完成并通过对应验证。
- 顾客端能独立完成订单查看、售后申请、进度跟踪、补充凭证、确认完成和评价。
- 管理员端能完成审核、驳回、要求补充、分派、处理记录、SLA 风险查看和客户历史查看。
- AI 只作为辅助层存在，关闭 AI 后主业务流程仍然可跑通。
- 售后申请、订单、工单、凭证、评价、AI 建议、处理日志能串成可审计证据链。
- 自动化验证覆盖主流程，新增改动不依赖手工猜测是否可用。
- 进度文档中没有未处理的 P0/P1 高价值问题；剩余 P2/P3 问题有明确原因、收益不足或后续扩展说明。
