## 总体开场

开场 20 秒：

> 老师好，我这个系统不是只做一个聊天框，而是一个退换货智能客服中台。客户可以咨询和提交售后，管理员可以审核、分派、跟进 SLA、接管工单。AI 在里面是增强层，负责意图识别、知识检索、图片风险预审和回复草稿，但真正的状态流转、权限控制、人工复核、数据库日志和本地兜底都由系统业务逻辑完成。

建议演示入口：管理员登录后打开 `/showcase`，不要单独讲 `/operations` 或 `/feature-closures`，它们已经重定向到售后运营首页。

## 8-10 分钟演示顺序

### 1. 售后运营首页

页面：`/showcase`

话术：

> 这里是售后运营首页，相当于答辩展示中心。可以看到系统不是单点功能，而是围绕售后场景拆成客户咨询、售后审核、SLA 跟进、人工工单、客户画像、商品质量预警和日志证据链几条线。
> 我后面会按真实客服处理顺序演示：客户先咨询，系统识别意图和订单上下文，然后上传凭证，管理员审核风险，最后进入工单、SLA 和运营分析。

可以强调：

- 客户侧、管理员侧都有权限隔离。
- AI 能力没有绕过业务规则。
- 首页只是导航和总览，真实数据都在后面的业务页面。

### 2. 咨询工作台：多轮上下文和知识库命中

页面：`/chat`

推荐绑定订单：`OPS202605120002`

打开页面时先看左侧会话列表：

> 我这里专门把重置脚本做成“干净演示库”。聊天列表保留的是 6 条典型售后场景，不是随机测试数据，也没有浏览器测试、上传调试、乱码标题这类脏会话。它们分别覆盖图片凭证预审、退款到账、多轮规则命中、退款催促、投诉转人工、物流异常和本地规则兜底。正式演示我重点讲第一条，其余几条可以快速点开证明覆盖面。

6 条会话怎么讲：

- `图片凭证风险预审-耳机断连换货`：主线，展示 AI 图片风险、C2PA、补证建议和工单。
- `退款多久到账-多轮规则命中`：展示上一轮订单语境和退款到账知识库命中。
- `退款催促-高优先级工单`：展示催促/投诉倾向如何形成高优先级任务。
- `投诉转人工-SLA 跟进`：展示人工接管和 SLA 排序。
- `物流 48 小时未更新-人工核实`：展示物流异常规则和承运商核实建议。
- `激活商品退货规则-本地兜底`：展示模型不可用时规则模板仍能稳定答复。

可输入三轮问题：

```text
这个耳机右耳总是断连，可以直接换新吗？
那我明天要出差，能不能快一点处理？
如果商家不处理，我能投诉吗？
```

话术：

> 第一轮系统会识别为换货申请，不是简单关键词回复，而是结合订单已签收、商品类型和售后规则，命中换货凭证要求。
> 第二轮虽然没有再说订单号，系统仍然承接上一轮耳机订单语境，知道用户是在催处理时效。
> 第三轮识别为投诉和人工转接场景，会把咨询升级成工单，这体现了多轮上下文、知识库命中和业务动作联动。

重点不要只说“AI 聊天”，要说：

- 意图识别：退货、换货、退款进度、物流异常、投诉转人工。
- 知识检索：回答来自规则文档，不是随便生成。
- 本地兜底：模型不可用时也能用规则模板稳定回复。
- 工单联动：投诉或异常不是停留在聊天框，会转为客服后台任务。

### 3. 上传风险图片：AI 图片真实性预审和 C2PA

页面：仍在 `/chat`

风险图片：`tmp\uploads\demo\ai-risk-evidence.png`

话术：

> 这里我上传一张演示凭证，图片右下角和文件名都有“豆包AI生成”的来源线索。系统不会武断地说这张图一定真或一定假，而是做风险预审：识别 AI 生成风险、真实性风险、篡改风险，并建议补充原始实拍、连续故障视频、商品序列号和包装配件照片。
> 图片面板里还展示 C2PA 内容凭证状态。C2PA 能帮助判断图片来源，例如有没有可信签名、提供方和生成器信息；但如果没有 C2PA，也不能证明图片一定真实或虚假，只能说明来源凭证缺失，需要结合视觉信号和人工复核。

边界说清楚：

> 我这里把功能定位成“风险预审 + 人工复核”，不是承诺 100% 鉴伪。这样更符合真实客服系统，也能避免 AI 误判直接影响用户权益。

## 老师追问回答

**问：你用了 AI，是不是主要就是调接口？**
答：不是。AI 只是增强层，主系统仍是 Spring Boot + Vue 3 + MySQL 的售后业务系统。订单、售后状态、权限、SLA、工单、审核、日志都在本地业务代码和数据库里完成。LangChain4j 负责把模型接入 Java 服务，类似 MyBatis 负责把数据库接入 Java 服务，但它不会替代业务规则。

**问：AI 图片识别能保证判断真假吗？**
答：不能保证 100%。我把它设计为风险预审，不是最终鉴定。系统会输出 AI 生成风险、篡改风险、C2PA 状态和补证建议，最终由客服结合原始实拍、视频和订单规则人工复核。

**问：C2PA 没检测到，是不是说明图片是假的？**
答：不是。C2PA 缺失只能说明没有可信来源凭证，不能单独证明真假。系统会把它作为一个信号，再结合文件名、水印、视觉模型、本地规则和人工复核。

**问：模型服务挂了怎么办？**
答：系统保留本地规则和模板兜底。日志里会记录 `SKIPPED` 或 `FAILED`，前端仍能稳定返回规则答案、生成工单、做售后审核。答辩现场即使网络或模型不可用，也不会影响核心业务演示。

**问：为什么要做 AI 草稿，而不是自动回复用户？**
答：售后涉及退款、换货和投诉，直接自动回复风险比较高。所以我设计成客服副驾驶：AI 给草稿，客服采纳或废弃，并把动作写入日志，既提高效率，又保留人工责任边界。

**问：这个系统和普通 CRUD 有什么区别？**
答：普通 CRUD 只是增删改查。这个系统有业务闭环：客户咨询、意图识别、知识检索、售后申请、凭证审计、风险评分、SLA 排序、人工工单、客户画像、商品质量预警和证据链日志，能够模拟真实售后运营流程。

**问：怎样证明演示可复现？**
答：我准备了 `tools\reset-demo-data.cmd`，双击会重新执行建表、基础数据和演示数据，并生成固定风险图片。`tools\demo-precheck.ps1` 会检查图片风险样本、凭证审计、风险评估、草稿、工单、商品预警和日志样本是否齐全。
