# 答辩用：系统功能源码链路讲解

这份文档是给我自己答辩时使用的，不是给老师看的论文式说明。我的讲法要从页面开始，一直讲到后端 Controller、Service、Mapper、MyBatis XML、数据库表，再讲数据怎么以 JSON 返回前端、前端怎么绑定到页面。老师如果问“这个功能你怎么做的”，我就按下面每个功能的链路回答。

项目主栈是 Vue 3 + Element Plus + Pinia + Axios + Spring Boot + MyBatis + MySQL + LangChain4j。这个项目不是前端写死假数据，核心数据都从数据库来；AI 也不是替代业务规则，而是作为售后客服辅助层，真正的权限校验、订单校验、售后状态流转、工单创建都在 Java Service 层完成。

## 1. 全系统通用请求链路

### 我整体是怎么设计前后端交互的

每个功能基本都走同一条链路：

```text
Vue 页面
  -> ref / reactive / v-model 保存页面参数
  -> web/src/api/*.js 封装 Axios 请求
  -> web/src/api/request.js 自动加 Authorization: Bearer <JWT>
  -> Spring Controller 接收 query/path/body/header
  -> ServiceImpl 做业务校验、权限判断、状态流转、AI 或本地兜底
  -> Mapper 接口调用 MyBatis XML
  -> XML 查询或更新 MySQL 表
  -> Mapper 返回 Java 对象
  -> Service 组装对象、PageResult 或 Map
  -> Controller 用 Result.success(data) 包成统一 JSON
  -> Axios 响应拦截器取出 result.data
  -> Vue 把 data 写回 ref/reactive，页面自动刷新
```

### 统一返回格式

后端统一返回类在 `server/src/main/java/com/user/returnsassistant/pojo/Result.java`。

- `Result.success()` 返回 `code=1, msg=success, data=null`。
- `Result.success(data)` 返回 `code=1, msg=success, data=业务数据`。
- `Result.error(msg)` 返回 `code=0, msg=错误信息, data=null`。

分页对象在 `server/src/main/java/com/user/returnsassistant/pojo/PageResult.java`：

- `total` 表示总条数。
- `rows` 表示当前页数据。

前端统一请求封装在 `web/src/api/request.js`。这里创建了 Axios 实例，`baseURL` 默认是 `/api`。请求拦截器从 `localStorage` 读取 `returns_assistant_token`，如果有 token，就放到请求头：

```text
Authorization: Bearer <JWT>
```

响应拦截器读取后端返回的 `code/msg/data`。如果 `code === 1`，页面拿到的就是 `result.data`，所以页面里不需要每次都写 `result.data.data`；如果失败，就弹出错误消息。如果错误信息里包含“登录”，还会清理本地 token 并跳转到 `/login`。

### 答辩时可以这样总述

老师如果问我“你这个项目的前后端是怎么连起来的”，我可以这样答：

> 我每个功能都不是只写页面，而是按完整链路做的。前端页面用 `ref`、`reactive` 和 `v-model` 保存用户输入，比如搜索关键字、状态筛选、表单字段。点击查询或提交时，页面调用 `web/src/api` 里的接口函数，这些函数底层走统一的 Axios。Axios 会自动带上 JWT。后端 Controller 接收参数后交给 Service，Service 负责业务规则，比如权限、订单归属、售后状态能不能流转、是否需要创建工单。Service 再调用 Mapper，Mapper 对应 MyBatis XML，XML 里写动态查询条件和数据库表关联。数据查出来后回到 Service，再由 Controller 用 `Result.success(data)` 包成 JSON 返回。前端响应拦截器取出 `data`，页面把数据写入 `ref` 或 `reactive`，表格、详情区、时间线就会自动刷新。

## 2. 登录、注册、JWT 和路由权限

### 源码位置

- 前端页面：`web/src/views/LoginView.vue`
- 前端状态：`web/src/stores/authStore.js`
- 前端接口：`web/src/api/authApi.js`
- 路由守卫：`web/src/router/index.js`
- 后端入口：`server/src/main/java/com/user/returnsassistant/controller/AuthController.java`
- 后端业务：`server/src/main/java/com/user/returnsassistant/service/impl/AuthServiceImpl.java`
- JWT 工具：`server/src/main/java/com/user/returnsassistant/utils/JwtTokenProvider.java`
- 权限拦截器：`server/src/main/java/com/user/returnsassistant/config/AuthInterceptor.java`
- 数据库表：`user_account`
- MyBatis XML：`server/src/main/resources/mapper/UserAccountMapper.xml`

### 前端页面怎么做

登录页 `LoginView.vue` 用一个登录卡片承载账号、密码、注册昵称、手机号、确认密码。页面里有一个 `mode` 控制当前是登录还是注册，`form` 是 `reactive` 对象，保存这些字段：

```text
username
password
confirmPassword
displayName
phone
```

输入框用 `v-model="form.username"`、`v-model="form.password"` 这类写法绑定，所以用户输入内容会实时进入 `form`。点击登录按钮会执行 `submit()`：

- 如果当前是登录模式，就调用 `authStore.login(form.username, form.password)`。
- 如果当前是注册模式，就调用 `authStore.register({...form})`。

`authStore` 在 `web/src/stores/authStore.js`，它用 Pinia 保存 `token`、`user`、`loading`。登录成功后执行 `saveAuth(data)`，把后端返回的 JWT 保存到 `localStorage` 的 `returns_assistant_token`，把用户信息保存到 `returns_assistant_user`。

### 前端怎么调后端

`web/src/api/authApi.js` 封装了 4 个接口：

```text
login(data) -> POST /auth/login
register(data) -> POST /auth/register
getMe() -> GET /auth/me
logout() -> POST /auth/logout
```

`request.js` 负责把 token 放到请求头，所以登录以后访问别的接口时，页面不需要每次手动加 token。

### 后端 Controller 怎么接参数

`AuthController` 类上有 `@RequestMapping("/auth")`。

- `login(@RequestBody LoginRequest request)` 接收 JSON 请求体，路径是 `POST /auth/login`。
- `register(@RequestBody RegisterRequest request)` 接收注册 JSON，路径是 `POST /auth/register`。
- `me(HttpServletRequest request)` 从请求头取 `Authorization`，路径是 `GET /auth/me`。
- `logout(HttpServletRequest request)` 也是从请求头取 token，路径是 `POST /auth/logout`。

Controller 不直接写数据库逻辑，而是调用 `AuthService`：

```text
authService.login(username, password)
authService.register(...)
authService.me(Authorization)
authService.logout(Authorization)
```

### Service 层具体做什么

`AuthServiceImpl.login` 先调用 `UserAccountMapper.getByUsername(username)` 查询 `user_account` 表。查到用户后，它会判断：

- 用户是否存在。
- 用户 `status` 是否启用。
- 密码是否匹配。

通过后调用 `JwtTokenProvider.createToken(...)` 生成 JWT。JWT 里面保存用户 ID、用户名、昵称、角色、签发时间、过期时间和 `jti`。后端返回 `LoginResponse`，里面有：

```text
token
userId
username
displayName
role
expiresAt
```

后续接口校验登录时，Service 会先去掉 `Bearer ` 前缀，再解析 JWT。解析成功后还会用 JWT 里的用户 ID 回查 `user_account` 表，确认账号仍然存在并且启用。这样即使 token 没过期，账号被停用后也不能继续访问。

退出登录时，`logout` 会读取 JWT 的 `jti`，放入撤销列表，后续同一个 token 再访问就会被拒绝。

### 路由权限怎么做

前端路由在 `web/src/router/index.js`。每个页面通过 `meta` 标记权限：

- `public: true` 表示登录页这类公开页面。
- `adminOnly: true` 表示管理员页面。
- `customerOnly: true` 表示顾客页面。

`router.beforeEach` 会判断：

- 没登录访问业务页，跳转 `/login`。
- 顾客访问管理员页，跳回 `/customer/after-sales`。
- 管理员访问顾客页，跳回 `/admin/after-sales/review`。
- 已登录访问 `/login`，按角色跳到默认首页。

后端管理员接口还会用 `@OperatorAnno` 标记。`AuthInterceptor` 检查到这个注解后，会调用 `authService.requireAdmin(...)`，确保只有管理员能访问。

### 我可以这样回答老师

> 登录这块我前端写在 `LoginView.vue`，用 `v-model` 把账号密码绑定到 `form`。点击登录后调用 Pinia 里的 `authStore.login`，再通过 `authApi.login` 请求 `/auth/login`。后端 `AuthController` 用 `@RequestBody LoginRequest` 接收账号密码，然后交给 `AuthServiceImpl`。Service 通过 `UserAccountMapper.getByUsername` 查 `user_account` 表，校验账号状态和密码，通过后用 `JwtTokenProvider` 生成 JWT。JWT 返回前端后，`authStore` 保存到 `localStorage`。之后 Axios 拦截器每次请求都会自动带 `Authorization: Bearer token`。前端路由守卫根据角色控制顾客页和管理员页，后端 `@OperatorAnno` 和 `AuthInterceptor` 再做一次管理员权限校验。

## 3. 顾客售后中心

顾客售后中心页面是 `web/src/views/CustomerAfterSaleCenterView.vue`，路由是 `/customer/after-sales`。这个页面是顾客端核心页面，包含我的订单、我的售后、申请售后、补充凭证、服务评价几个功能。

### 3.1 顾客查询自己的订单

#### 源码位置

- 前端页面：`CustomerAfterSaleCenterView.vue`
- 前端 API：`web/src/api/orderApi.js`
- 后端 Controller：`OrderController.page`
- 后端 Service：`OrderServiceImpl.page`
- Mapper XML：`DemoOrderMapper.xml`
- 数据库表：`demo_order`

#### 前端怎么布局和收集参数

页面左侧有“我的订单”区域，包含搜索框、订单表格、分页和申请按钮。前端定义：

```text
orderQuery = reactive({ page: 1, pageSize: 8, keyword: '' })
orders = ref([])
orderTotal = ref(0)
selectedOrder = ref(null)
```

搜索框绑定 `orderQuery.keyword`，分页绑定 `orderQuery.page` 和 `orderQuery.pageSize`。点击查询时调用 `loadOrders()`。

#### 参数怎么传后端

`loadOrders()` 调用：

```text
pageOrders(orderQuery)
```

`orderApi.js` 里对应：

```text
pageOrders(params) -> GET /orders
```

Axios 会把 `orderQuery` 变成 query 参数，例如：

```text
GET /orders?page=1&pageSize=8&keyword=鞋子
```

#### 后端怎么处理

`OrderController.page(OrderSearch search, HttpServletRequest request)` 接收查询参数。`OrderSearch` 里有 `page`、`pageSize`、`keyword`、`orderStatus`、`logisticsStatus`、`afterSaleStatus` 等字段。

Controller 会通过 token 获取当前用户。如果当前是顾客，就把 `search.userId` 强制设置成当前顾客 ID，保证顾客只能查自己的订单；管理员才可以查全部订单。

然后 Controller 调用 `orderService.page(search)`。`OrderServiceImpl.page` 里先执行：

```text
PageHelper.startPage(search.getPage(), search.getPageSize())
orderMapper.page(search)
```

`DemoOrderMapper.xml` 从 `demo_order` 表查询。`keyword` 会匹配订单号、商品名等字段，最后按更新时间和 ID 倒序返回。PageHelper 自动处理分页总数和当前页数据。

#### 数据怎么回到页面

后端返回 `PageResult<DemoOrder>`，Controller 包成 `Result.success(data)`。前端 Axios 拦截器取出 `data` 后，页面执行：

```text
orders.value = data.rows || []
orderTotal.value = data.total || 0
```

表格通过 `:data="orders"` 渲染订单，分页组件用 `orderTotal` 显示总数。点击某一行会把这行订单写入 `selectedOrder`，后面申请售后时就使用这条订单。

#### 我可以这样回答老师

> 顾客订单查询是从 `CustomerAfterSaleCenterView.vue` 的“我的订单”区域开始的。页面用 `orderQuery` 保存页码、每页条数和搜索关键字，输入框通过 `v-model` 绑定 `keyword`。点击查询后调用 `orderApi.pageOrders` 请求 `/orders`。后端 `OrderController.page` 用 `OrderSearch` 接收 query 参数，然后根据 JWT 判断当前用户，如果是顾客就把 `userId` 限制为当前用户，防止查到别人的订单。Service 层用 PageHelper 分页，再调用 `DemoOrderMapper.page`，MyBatis XML 从 `demo_order` 表查询。返回后前端把 `rows` 写入 `orders`，把 `total` 写入 `orderTotal`，Element Plus 表格就显示出来。

### 3.2 顾客提交售后申请

#### 源码位置

- 前端页面：`CustomerAfterSaleCenterView.vue`
- 前端 API：`customerAfterSaleApi.js`
- 后端 Controller：`CustomerAfterSaleController.create`
- 后端 Service：`AfterSaleApplicationServiceImpl.create`
- Mapper XML：`AfterSaleApplicationMapper.xml`、`AfterSaleProcessLogMapper.xml`、`DemoOrderMapper.xml`
- 数据库表：`after_sale_application`、`after_sale_process_log`、`demo_order`

#### 前端怎么布局和收集参数

顾客在订单表格里选中订单后，点击“申请售后”，页面执行 `openApplyDialog(order)` 打开弹窗。弹窗里有：

- 售后类型单选：`RETURN`、`EXCHANGE`、`REFUND`、`COMPLAINT`
- 原因类型下拉框
- 退款金额输入框
- 问题说明文本框

这些字段保存在：

```text
applyForm = reactive({
  serviceType: 'RETURN',
  reasonCode: '',
  reasonText: '',
  refundAmount: 0
})
```

提交时 `submitApplication()` 组装请求体：

```text
orderId: applyingOrder.value.id
serviceType: applyForm.serviceType
reasonCode: applyForm.reasonCode
reasonText: applyForm.reasonText
refundAmount: applyForm.refundAmount
```

#### 参数怎么传后端

前端调用：

```text
createCustomerAfterSale(data) -> POST /customer/after-sales
```

这是 JSON 请求体，所以后端用 `@RequestBody` 接收。

#### 后端怎么处理

`CustomerAfterSaleController.create` 先调用 `authService.requireCustomer(...)`，确认当前用户是顾客。然后把请求交给：

```text
afterSaleApplicationService.create(createRequest, user)
```

`AfterSaleApplicationServiceImpl.create` 做了很多真实业务规则：

- 必须选择订单。
- 订单必须存在，调用 `orderMapper.getById(orderId)`。
- 顾客只能给自己的订单申请售后。
- 未支付订单不能申请售后。
- 待支付或关闭订单不能申请售后。
- 同一订单不能已有进行中的售后，调用 `applicationMapper.countActiveByOrderId(orderId)`。
- 售后类型只能是 `RETURN`、`EXCHANGE`、`REFUND`、`COMPLAINT`。
- 退款金额不能小于 0，也不能超过订单金额。

通过后，Service 创建 `AfterSaleApplication` 对象：

- `applicationNo` 用 `NoUtils.afterSaleNo()` 生成。
- `status` 初始为 `SUBMITTED`。
- `priority` 根据投诉类型或金额生成。
- `riskLevel` 根据投诉类型或金额生成。
- `slaDeadline` 投诉是 12 小时，普通售后是 48 小时。

然后执行：

```text
applicationMapper.insert(application)
writeLog(..., "SUBMIT", null, "SUBMITTED", ...)
orderMapper.updateAfterSaleStatus(...)
return getById(application.getId())
```

`AfterSaleApplicationMapper.xml` 把主表数据插入 `after_sale_application`。`AfterSaleProcessLogMapper.xml` 插入流程日志到 `after_sale_process_log`。`DemoOrderMapper.xml` 更新 `demo_order.after_sale_status`。

#### 数据怎么回到页面

Service 返回完整售后详情，里面有主表数据、处理日志、凭证等。前端提交成功后关闭弹窗，然后调用 `reloadAll()` 同时刷新订单列表和售后列表。这样老师看数据库时，能看到主表、日志表、订单状态都被同步更新。

#### 我可以这样回答老师

> 顾客提交售后不是只插一条数据。前端在售后申请弹窗里用 `applyForm` 保存售后类型、原因、说明和退款金额，同时把选中订单的 `orderId` 一起提交到 `/customer/after-sales`。后端 `CustomerAfterSaleController` 先校验当前用户必须是顾客，然后交给 `AfterSaleApplicationServiceImpl.create`。Service 先查 `demo_order`，判断订单是不是当前顾客的、是否已支付、状态是否允许售后、有没有进行中的售后、金额是否超过订单金额。通过后插入 `after_sale_application`，再写 `after_sale_process_log`，并更新 `demo_order.after_sale_status`。最后返回完整详情，前端刷新订单和售后列表。

### 3.3 顾客查看售后列表和详情

#### 源码位置

- 前端页面：`CustomerAfterSaleCenterView.vue`
- 前端 API：`pageCustomerAfterSales`、`getCustomerAfterSale`
- 后端 Controller：`CustomerAfterSaleController.page`、`getById`
- 后端 Service：`AfterSaleApplicationServiceImpl.page`、`getById`
- Mapper XML：`AfterSaleApplicationMapper.xml`、`AfterSaleProcessLogMapper.xml`、`AfterSaleEvidenceMapper.xml`
- 数据库表：`after_sale_application`、`after_sale_process_log`、`after_sale_evidence`

#### 前端怎么布局和收集参数

页面右侧是“我的售后”区域。查询条件保存在：

```text
afterSaleQuery = reactive({ page: 1, pageSize: 8, status: '' })
afterSales = ref([])
afterSaleTotal = ref(0)
selectedAfterSale = ref(null)
```

状态下拉框绑定 `afterSaleQuery.status`，点击查询调用 `loadAfterSales()`。

#### 后端怎么处理

前端请求：

```text
GET /customer/after-sales?page=1&pageSize=8&status=SUBMITTED
```

`CustomerAfterSaleController.page` 从 JWT 里取当前顾客，并强制 `search.userId = customer.id`。然后 Service 用 PageHelper 调用 `applicationMapper.page(search)`。

`AfterSaleApplicationMapper.xml` 的 `BaseSelect` 不只查主表，还关联：

```text
after_sale_application a
left join demo_order o on a.order_id=o.id
left join user_account u on a.user_id=u.id
left join user_account assignee on a.assigned_to=assignee.id
left join service_ticket t on a.ticket_id=t.id
```

所以列表能显示售后单号、订单号、商品名、用户昵称、指派人、关联工单等信息。

查看详情时，前端调用：

```text
GET /customer/after-sales/{id}
```

Controller 先取详情，再用 `authService.ensureSelfOrAdmin(...)` 确保顾客只能看自己的售后。Service 的 `getById` 会补充：

- `processLogs = processLogMapper.listByApplicationId(id)`
- `evidences = evidenceMapper.listByApplicationId(id)`
- `customerResultSummary`
- `customerFinalReply`
- `customerNextAction`

#### 我可以这样回答老师

> 售后列表查询和订单查询类似，前端用 `afterSaleQuery` 保存分页和状态筛选，调用 `/customer/after-sales`。后端 Controller 不相信前端传来的用户 ID，而是从 JWT 解析当前顾客，然后强制设置 `search.userId`。Service 用 PageHelper 分页，Mapper XML 从 `after_sale_application` 主表查，还关联订单表、用户表和工单表。查看详情时，Service 会继续查处理日志表和凭证表，并组装顾客能看懂的结果说明和下一步动作。

### 3.4 顾客补充凭证

#### 源码位置

- 前端页面：`CustomerAfterSaleCenterView.vue`
- 前端 API：`addCustomerAfterSaleEvidence`
- 后端 Controller：`CustomerAfterSaleController.addEvidence`
- 后端 Service：`AfterSaleApplicationServiceImpl.addEvidence`
- Mapper XML：`AfterSaleEvidenceMapper.xml`、`AfterSaleProcessLogMapper.xml`
- 数据库表：`after_sale_evidence`、`after_sale_process_log`

#### 链路说明

前端打开补充凭证弹窗，表单对象是：

```text
evidenceForm = reactive({
  evidenceType: 'TEXT',
  fileUrl: '',
  content: ''
})
```

提交时调用：

```text
POST /customer/after-sales/{id}/evidence
```

后端 `CustomerAfterSaleController.addEvidence` 用 `@PathVariable Long id` 接收售后申请 ID，用 `@RequestBody AfterSaleEvidenceRequest` 接收凭证内容。

Service 会先 `getById(id)` 查售后申请，再判断：

- 顾客只能给自己的售后申请补材料。
- 终态申请不能补材料，比如 `REJECTED`、`COMPLETED`、`CANCELLED`。
- 凭证类型只能是 `IMAGE`、`VIDEO`、`TEXT`、`LOGISTICS_NO`。
- 凭证内容不能为空。

通过后插入 `after_sale_evidence`，同时写一条 `SUPPLEMENT_EVIDENCE` 流程日志。

#### 我可以这样回答老师

### 4.3 审核通过、驳回、要求补材料、确认完成

#### 源码位置

#### 后端业务规则

`approve` 会检查当前状态是否可审核，然后确定通过后的下一状态：

- `REFUND` -> `REFUNDING`
- `EXCHANGE` -> `EXCHANGING`
- `COMPLAINT` -> `APPROVED`
- 默认退货 -> `WAIT_BUYER_SEND`

`reject` 要求必须填写驳回原因，然后状态改成 `REJECTED`，写 `REJECT` 日志，订单售后状态改成 `REJECTED`。

`complete` 要求申请不能已经结束，也不能还处于待审核状态。完成后状态改成 `COMPLETED`，写 `CONFIRM` 日志，订单售后状态改成 `FINISHED`。

#### 我可以这样回答老师

> 管理员审核时前端只有一个 `decisionForm`，里面保存通过金额和备注。不同按钮调用不同后端接口，比如通过是 `/approve`，驳回是 `/reject`，补材料是 `/request-evidence`，完成是 `/complete`。后端 `AfterSaleApplicationServiceImpl` 会先判断当前售后状态能不能操作。比如驳回必须写原因，通过金额不能超过申请金额，完成前不能还停留在审核状态。每次操作都会更新 `after_sale_application` 的状态和金额，同时写入 `after_sale_process_log`，还会同步更新订单表的售后状态。

### 5.2 发送消息和 AI 回复

#### 源码位置

- 前端页面：`ChatWorkbenchView.vue`
- 前端 API：`sendMessageStream`、`sendMessage`
- 后端 Controller：`ChatSessionController.sendMessageStream`、`sendMessage`
- 后端 Service：`ChatServiceImpl.sendMessage`
- AI Service：`AiServiceImpl`
- LangChain4j 工具：`AiBusinessToolServiceImpl`
- Mapper XML：`ChatMessageMapper.xml`、`IntentRecordMapper.xml`、`RetrievalLogMapper.xml`、`AiCallLogMapper.xml`、`ProcessTraceMapper.xml`、`KnowledgeDocMapper.xml`
- 数据库表：`chat_message`、`intent_record`、`retrieval_log`、`ai_call_log`、`process_trace`、`knowledge_doc`

#### 前端怎么传

页面输入框绑定 `draft`，开关绑定 `useAi`，订单号绑定 `orderNo`。点击发送执行 `submit()`，组装：

```text
content: draft.value
orderNo: orderNo.value
useAi: useAi.value
```

优先调用流式接口：

```text
sendMessageStream(id, data, handlers)
POST /chat-sessions/{id}/message-stream
```

这个函数底层用 `fetch`，手动设置：

```text
Content-Type: application/json
Authorization: Bearer <token>
```

前端读取 SSE 事件：

- `progress` 更新页面进度。
- `final` 接收最终数据。
- `error` 显示异常。

#### 后端怎么处理

`ChatSessionController.sendMessageStream` 先校验会话权限和订单权限，然后创建 `SseEmitter`。它先发送几个进度事件，比如“消息已发送”“正在识别意图、读取订单并检索知识库”，然后真正调用：

```text
chatService.sendMessage(id, messageRequest)
```

`ChatServiceImpl.sendMessage` 的核心流程是：

1. 查会话，判断会话不能是 `CLOSED`。
2. 校验消息内容不能为空。
3. 根据订单号解析订单，必要时把订单绑定到会话。
4. 查询最近 8 条消息，构造多轮上下文。
5. 计算下一条消息序号，把用户消息插入 `chat_message`。
6. 写 `CONTEXT_RESOLVE` 处理轨迹。
7. 用规则识别意图，插入 `intent_record`。
8. 构造订单上下文，写 `ORDER_CONTEXT` 轨迹。
9. 用 `KnowledgeDocMapper.search` 检索知识库，命中结果写入 `retrieval_log`。
10. 根据意图、订单和知识库生成本地兜底回复。
11. 调用 `AiBusinessToolServiceImpl` 获取订单状态、知识库证据和工单工具证据。
12. 构造 prompt，调用 `aiService.generate(prompt)`。
13. 把 AI 调用结果写入 `ai_call_log`。
14. 如果 AI 成功，用 AI 回复；否则用本地兜底回复。
15. 插入客服回复到 `chat_message`。
16. 判断是否需要转人工，必要时创建或复用 `service_ticket`。
17. 更新会话摘要和当前意图。
18. 返回用户消息、客服消息、意图、上下文、知识命中、AI 状态、工单和处理轨迹。

#### MyBatis 和数据库怎么配合

- `ChatMessageMapper.xml` 插入用户消息和客服回复，按 `seq_no` 排序查询。
- `IntentRecordMapper.xml` 保存识别到的意图、置信度和方法。
- `KnowledgeDocMapper.xml` 从 `knowledge_doc` 按标题、问题、正文、关键词或意图检索。
- `RetrievalLogMapper.xml` 保存每次知识命中的文档、排名、分数和命中原因。
- `AiCallLogMapper.xml` 保存模型、状态、耗时、token 和错误信息。
- `ProcessTraceMapper.xml` 保存每一步处理轨迹。

#### 前端怎么展示返回

最终返回的 `finalData` 里有：

```text
userMessage
assistantMessage
intent
context
orderContext
knowledgeHits
businessTools
ai
ticket
trace
suggestedQuestions
```

前端把消息追加到消息区，把意图、知识命中、AI 状态、流程轨迹等写入右侧面板。由于 Vue 是响应式，`messages`、`traces`、`ticket` 等对象更新后页面自动刷新。

#### 我可以这样回答老师

> 聊天功能是我项目里最长的一条链路。前端输入框用 `draft` 保存用户问题，订单号用 `orderNo` 保存，是否启用 AI 用 `useAi` 保存。点击发送后请求 `/chat-sessions/{id}/message-stream`，前端用 SSE 接收进度。后端先校验会话和订单权限，然后 `ChatServiceImpl.sendMessage` 会插入用户消息、解析多轮上下文、识别意图、查询订单、检索知识库、记录检索日志、构造本地兜底回复，再调用 LangChain4j 相关 AI 服务生成增强回复。无论 AI 成功还是失败，系统都会保存 `ai_call_log` 和 `process_trace`。最后把客服回复插入 `chat_message`，必要时创建人工工单。前端拿到最终 JSON 后把消息、意图、知识命中、AI 状态和轨迹绑定到页面。

### 5.3 人工工单转接

聊天里如果识别到投诉、人工客服、物流异常等场景，`ChatServiceImpl.handleTicketHandoff` 会调用：

```text

```

`ServiceTicketServiceImpl.createFromSession` 会先看当前会话有没有未关闭工单，如果有就复用；没有才插入新的 `service_ticket`。优先级根据意图、用户问题和订单物流状态判断：

- 投诉或人工转接 -> `HIGH`
- 物流异常 -> `HIGH`
- 丢失、损坏、严重 -> `URGENT`
- 其他 -> `NORMAL`

这样 AI 客服不能处理的场景，会自动进入人工工单闭环。

后端 `ChatServiceImpl.buildEvidenceReport` 会汇总：

- 会话信息
- 订单上下文
- 对话记录
- 意图识别记录
- 知识命中记录
- AI 调用日志
- 人工工单
- 处理轨迹

最后返回 Markdown 文本。这个功能是为了答辩时证明系统有可追踪证据，不是只返回一段 AI 文本。

XML 的 `search` 从 `knowledge_doc` 查询启用文档，支持按意图过滤，按优先级和更新时间排序。

### 我可以这样回答老师

## 16. 后端分层结构怎么讲

我可以把后端解释成四层：

### Controller 层

Controller 负责接收 HTTP 请求，常见参数来源有：

- `@RequestBody`：接收 JSON 请求体，例如登录、注册、售后申请、审核动作。
- `@PathVariable`：接收路径参数，例如 `/customer/after-sales/{id}`。
- 普通对象参数：接收 query 参数，例如 `OrderSearch search`、`AfterSaleApplicationSearch search`。
- `HttpServletRequest`：读取 `Authorization` 请求头。

Controller 不直接写 SQL，主要负责把参数交给 Service，并用 `Result.success(...)` 返回。

### Service 层

Service 是业务核心，负责：

- 登录和 JWT 解析。
- 顾客和管理员权限。
- 顾客只能访问自己的订单、售后和会话。
- 售后状态流转。
- 退款金额、订单状态、重复售后校验。
- AI 生成和本地兜底。
- 工单优先级和人工转接判断。
- 统计看板和日志诊断。

### Mapper 层

Mapper 是 Java 接口，对应 MyBatis XML。Service 调用 Mapper 方法，不直接拼 SQL。

### MyBatis XML 和数据库

XML 负责具体 SQL：

- 动态 `where` 条件。
- `left join` 关联表。
- `insert/update/delete`。
- 排序。

主要表包括：

| 模块 | 数据表 |
| --- | --- |
| 用户和权限 | `user_account` |
| 订单 | `demo_order` |
| 顾客售后 | `after_sale_application`、`after_sale_evidence`、`after_sale_process_log`、`service_review` |
| 聊天和 AI | `chat_session`、`chat_message`、`intent_record`、`retrieval_log`、`ai_call_log`、`process_trace` |
| 知识库 | `knowledge_category`、`knowledge_doc` |
| 工单 | `service_ticket`、`reply_draft` |

## 17. 一条完整例子：顾客申请售后

老师如果让我完整讲一条链路，我最适合讲“顾客申请售后”，因为它包含页面、接口、权限、业务规则、数据库和返回展示。

### 完整链路

1. 顾客进入 `/customer/after-sales`，页面加载时调用 `loadOrders()` 查询自己的订单。
2. 前端 `orderQuery` 保存页码和关键字，请求 `GET /orders`。
3. 后端 `OrderController.page` 根据 JWT 限制 `userId`，Service 查 `demo_order`。
4. 前端把 `rows` 绑定到订单表格。
5. 顾客选中订单，点击“申请售后”。
6. 页面打开弹窗，`applyForm` 保存售后类型、原因、说明和退款金额。
7. 点击提交，前端调用 `POST /customer/after-sales`，请求体包含 `orderId`、`serviceType`、`reasonCode`、`reasonText`、`refundAmount`。
8. 后端 `CustomerAfterSaleController.create` 用 `@RequestBody` 接收请求，用 JWT 获取当前顾客。
9. `AfterSaleApplicationServiceImpl.create` 查订单，判断订单归属、支付状态、订单状态、重复售后、退款金额和售后类型。
10. Service 创建售后申请，插入 `after_sale_application`。
11. Service 写入 `after_sale_process_log`，记录 `SUBMIT`。
12. Service 更新 `demo_order.after_sale_status`。
13. Service 再查详情，把流程日志和凭证一起带回。
14. Controller 用 `Result.success(application)` 返回 JSON。
15. 前端 Axios 拿到 `data`，关闭弹窗，调用 `reloadAll()`。
16. 页面重新加载订单和售后列表，顾客能看到刚提交的售后单。

### 我可以这样完整回答

> 比如顾客申请售后这条链路，我是从前端页面开始做的。顾客先在 `CustomerAfterSaleCenterView.vue` 查自己的订单，查询条件保存在 `orderQuery`，通过 `orderApi.pageOrders` 请求 `/orders`。后端 `OrderController` 根据 JWT 把查询限制到当前顾客，Service 通过 MyBatis 查 `demo_order` 表，返回 `PageResult`，前端把 `rows` 绑定到订单表格。顾客选中订单后打开售后弹窗，`applyForm` 保存售后类型、原因、说明和退款金额。提交时前端把这些字段加上 `orderId` 作为 JSON 发到 `/customer/after-sales`。后端 `CustomerAfterSaleController` 用 `@RequestBody` 接收，先确认当前用户是顾客，再交给 `AfterSaleApplicationServiceImpl.create`。Service 查订单，校验订单归属、支付状态、订单状态、是否已有进行中的售后、退款金额是否超限。通过后插入 `after_sale_application`，写入 `after_sale_process_log`，并更新 `demo_order.after_sale_status`。最后 Controller 用 `Result.success` 返回完整售后详情，前端刷新列表，页面就能看到新申请。

## 18. 老师可能追问的问题

### Q1：为什么说这是前后端分离系统？

可以答：

> 因为前端 Vue 只负责页面展示和交互，后端 Spring Boot 负责业务和数据。前端所有数据都通过 `web/src/api/*.js` 请求后端接口，后端用 `Result` 返回 JSON。比如订单、售后、聊天、知识库、工单都不是写死在前端，而是从后端接口查数据库返回。

### Q2：前端参数怎么传到后端？

可以答：

> 查询类参数一般放在 query 里，比如 `GET /orders?page=1&pageSize=10&keyword=...`，前端用 `request.get(url, { params })`。新增或操作类参数放在 JSON body 里，比如登录、售后申请、审核动作，前端用 `request.post(url, data)`。路径里的 ID 用 `/xxx/{id}`，后端用 `@PathVariable` 接收。

### Q3：Controller 层具体写了什么？

可以答：

> Controller 主要负责 HTTP 入口，比如 `@GetMapping`、`@PostMapping`、`@PutMapping`、`@DeleteMapping`。它接收参数后不直接查数据库，而是调用 Service。最后统一用 `Result.success(data)` 返回。需要鉴权的接口会从 `HttpServletRequest` 里取 `Authorization`，或者用 `@OperatorAnno` 让拦截器校验管理员。

### Q4：Service 层为什么重要？

可以答：

> Service 层是业务核心。比如售后申请不是直接插表，Service 要判断订单是不是当前顾客的、是否已支付、是否已经有进行中的售后、金额是否超过订单金额。审核时还要判断状态能不能流转。聊天时 Service 要识别意图、查订单、检索知识库、调用 AI、写日志和创建工单。所以业务规则主要都在 Service 层。

### Q5：Mapper 和 MyBatis XML 有什么作用？

可以答：

> Mapper 是 Java 调数据库的接口，XML 写具体 SQL。比如 `AfterSaleApplicationMapper.page` 对应 XML 里的 `select id="page"`，它从售后主表查，还关联订单、用户和工单。这样 Service 不需要拼 SQL，只调用 Mapper 方法，SQL 统一放在 XML 里维护。

### Q6：JSON 返回前端后怎么显示到页面？

可以答：

> 后端返回 `Result.success(data)`，前端 Axios 拦截器会把 `data` 取出来。分页数据一般有 `rows` 和 `total`，页面把 `rows` 赋值给表格数组，比如 `orders.value = data.rows`，把 `total` 赋给分页总数。详情数据赋给 `selected`。因为 Vue 是响应式，变量变化后表格、详情区、时间线都会自动更新。

### Q7：AI 失败时系统还能不能用？

可以答：

> 能用。AI 是增强层，不是业务系统的唯一依赖。聊天里 `ChatServiceImpl` 会先生成本地规则回复，再尝试调用 AI。如果 AI 成功就用 AI 回复；如果失败或没启用，就用本地兜底回复。回复草稿也是一样，AI 不可用时保存本地模板。所有失败都会写入 `ai_call_log`，方便日志中心诊断。

### Q8：怎么证明不是演示页面？

可以答：

> 可以从三方面证明。第一，所有页面都有对应接口和数据库表，比如订单表、售后主表、凭证表、流程日志表、聊天消息表、工单表。第二，关键操作会写多张表，比如售后申请会写主表、日志表并更新订单状态；聊天会写消息、意图、检索日志、AI 日志和处理轨迹。第三，日志中心和证据报告能把 AI 调用、知识命中、流程轨迹查出来，说明不是前端写死的效果。
