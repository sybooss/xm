# 开题答辩讲稿（约 5-8 分钟）

各位老师好，我的开题题目是《基于检索增强的电商退换货智能客服系统设计与实现》。

首先我想说明一下选题背景。随着电商业务规模不断扩大，用户在售后阶段会频繁提出退货、换货、退款进度、物流异常和人工投诉等问题。这类问题看似常见，但实际上涉及订单状态、商品类别、签收时间、退款路径和平台规则等多种约束。如果完全依赖人工客服处理，不仅成本较高，而且在高峰时段还容易出现响应慢、标准不统一和用户满意度下降的问题。另一方面，虽然通用大模型具备较强的语言生成能力，但在电商售后这种规则较强的场景中，如果缺少知识约束，系统就容易出现回答不稳或者依据不足的问题。因此，我选择把研究对象聚焦在电商退换货场景，设计一个面向领域规则的智能客服系统。

从国内外研究现状来看，当前相关研究主要集中在三个方向。第一类是任务型对话系统，重点研究多轮对话管理与语义理解；第二类是知识增强问答和检索增强生成，强调外部知识接入和回答可追溯；第三类是中文客服场景数据与行业应用研究，尤其是在电商客服、多轮中文对话和商品问答方面已经有一定基础。现有方法总体上有两个明显问题：一是纯规则系统可控但不够灵活，二是纯生成系统自然但不够稳定。所以本项目采用折中方案，也就是把意图识别、知识库检索、订单上下文和回复生成结合起来，构建一个既能回答问题、又能返回规则依据的领域智能客服系统。

本项目的目标，是设计并实现一个面向电商退换货场景的客服原型系统。围绕这个目标，我主要开展五项工作。第一是售后问题意图分类，把用户问题划分为售前咨询、退货申请、换货申请、退款进度、物流查询、规则说明、投诉或人工转接这七类。第二是构建退换货知识库，把常见售后规则、退款说明和异常处理流程整理成可检索文档。第三是实现多轮对话管理，使系统能够承接上下文，而不是每次都把用户问题当成新的对话。第四是把订单信息引入回答过程，例如订单号、商品名称、订单状态和下单时间，让回答更贴近真实业务场景。第五是完成系统前后端原型与测试验证，确保项目不仅能写出来，而且能跑起来、讲清楚。

在技术路线方面，本项目采用的核心链路是：前端交互到 FastAPI 接口层，再到意图识别、知识库检索、订单上下文融合、回复生成，最后返回回答依据与执行轨迹。具体来说，用户首先在前端输入问题，系统读取当前会话历史，再由意图识别模块判断当前问题属于哪一类客服意图。之后，系统根据问题内容和意图标签到本地知识库中进行检索，召回最相关的规则文档，并结合订单上下文组织最终回复。为了兼顾表达自然性和演示稳定性，系统同时支持两种回复模式：当配置了 OpenAI 接口时，使用大模型生成更自然的答案；当没有配置时，则回退到本地规则模板，保证系统在答辩现场也能稳定运行。

目前，这个项目已经具备较好的原型基础。后端部分已经基于 FastAPI 完成了系统状态、会话创建、消息处理和知识库管理等接口；前端部分已经完成了客服对话台和知识库中台两个页面；数据层采用 SQLite 保存知识文档和聊天记录；系统中已经实现了七类意图识别、30 多条售后规则知识文档、本地检索和双模式回复生成；同时还准备了演示脚本和测试用例，当前关键流程测试已经通过。这说明本项目不是从零开始设想，而是已经具备可行的系统基础，后续的重点是继续完善文档、优化模块细节并完成正式答辩材料。

本项目的创新点主要有三点。第一，它是面向电商退换货场景的领域化客服系统，不是泛化聊天机器人。第二，它把检索结果、订单状态和历史对话同时纳入回复生成过程，提高回答的针对性和可依据性。第三，它不仅能回答问题，还支持知识库管理与扩展，因此具备一定的工程可维护性和落地价值。

本项目的难点主要有三个。第一，相似售后问题之间的意图区分比较困难，比如退货和退款、物流异常和投诉之间常常存在交叉。第二，知识检索命中率和最终回答准确性之间需要平衡，既要找得准，也要说得清。第三，多轮上下文承接和现场演示稳定性需要同时保证，这对系统设计提出了更高要求。

在进度安排上，我会按照课程时间节点继续推进：前期完成开题报告和答辩材料，中期继续完善系统功能和测试验证，后期为结项验收准备更完整的演示和文档。总体来说，我希望最终完成一个能够真实展示“意图识别 + 知识检索 + 上下文融合 + 双模式回复生成”链路的电商退换货智能客服系统。
