# 新框架实施计划：电商退换货智能客服系统

## 1. 项目定位

本项目面向《复杂软件系统实践》课程，建设一个用于电商退换货场景的智能客服系统。系统不做通用聊天机器人，而是围绕退货申请、换货申请、退款进度、物流异常、规则说明、投诉与人工转接等典型售后问题，完成可演示、可维护、可验收的业务系统原型。

新版系统采用 `Vue 3 + Spring Boot + MySQL + LangChain4j` 技术路线：

- `Vue 3` 负责咨询工作台、知识库管理页、测试调试页和系统展示界面。
- `Spring Boot` 作为业务核心，负责 REST 接口、业务规则、会话管理、订单上下文、权限与日志。
- `MySQL` 负责结构化保存订单数据、知识文档、会话记录、消息记录、检索日志和 AI 调用日志。
- `LangChain4j` 作为 AI 增强层，负责提示词编排、上下文记忆、知识增强问答和大模型调用封装。

整体原则是“业务系统为主，AI 能力为辅”：退换货规则、订单状态判断、异常兜底和核心流程由 Spring Boot 控制；LangChain4j 用来增强语义理解和回答组织，但不能替代业务判断。

## 2. 建设目标与验收标准

### 2.1 建设目标

1. 完成一套前后端分离的电商退换货智能客服系统原型。
2. 支持用户在咨询工作台发起售后问题，并获得包含意图、依据、处理建议和追问提示的回复。
3. 支持知识库文档的新增、编辑、删除、查询、分类管理和检索调试。
4. 支持多轮会话，能够根据历史消息和订单上下文回答连续追问。
5. 支持 LangChain4j 增强回复，同时保留本地规则兜底，保证演示稳定。
6. 形成可用于结项展示的系统截图、测试用例、演示脚本和技术说明材料。

### 2.2 验收标准

- 系统能够通过本地启动方式运行：后端 Spring Boot 服务、前端 Vue 服务、MySQL 数据库均可连通。
- 内置不少于 30 条售后规则或 FAQ 文档，覆盖退货、换货、退款、物流、规则、投诉等场景。
- 支持不少于 7 类售后意图识别：售前咨询、退货申请、换货申请、退款进度、物流查询、规则说明、投诉与人工转接。
- 至少完成 10 组典型测试对话，其中包含单轮咨询、多轮追问、订单状态变化、知识库命中、AI 异常兜底等情况。
- 前端页面能够展示回答内容、命中知识、订单上下文、意图标签和处理轨迹。
- 后端能够记录会话消息、检索结果、AI 调用状态和异常信息，方便答辩时说明系统可维护性。

## 3. 总体架构设计

### 3.1 分层结构

系统按四层组织：

1. 表现层：Vue 3 前端页面，包含咨询工作台、知识库后台、对话测试、系统状态展示。
2. 业务层：Spring Boot Controller、Service、Domain 模块，处理接口编排、规则判断、会话流程和订单上下文。
3. 数据层：MySQL 数据库，保存知识文档、订单样例、会话消息、检索日志、AI 调用日志。
4. AI 增强层：LangChain4j 服务组件，负责提示词模板、上下文装配、知识增强生成和模型调用。

后续实现时建议在当前项目下新增新版工程目录：

## 4. 核心模块规划

### 4.1 前端模块

咨询工作台：

- 左侧展示会话列表和新建会话入口。
- 中间展示聊天消息、回复依据、处理建议和追问提示。
- 右侧展示订单上下文、意图识别结果、命中知识和系统处理轨迹。

知识库管理：

- 支持 FAQ 和规则文档的新增、编辑、删除、启用、停用。
- 支持按分类、关键词、适用场景、更新时间筛选。
- 提供检索调试入口，方便查看用户问题命中的知识内容。

系统测试与展示：

- 提供预置案例按钮，例如“退货申请”“退款多久到账”“物流异常投诉”等。
- 提供系统状态卡片，展示后端、数据库、AI 增强层是否可用。

### 4.2 后端业务模块

会话管理模块：

- 创建会话、保存消息、查询历史、维护多轮上下文。
- 为后续追问提供最近对话摘要和当前订单状态。

意图识别模块：

- 先用规则和关键词完成稳定识别。
- 在需要时调用 LangChain4j 辅助判断相似意图或复合意图。
- 输出统一意图标签、置信度、关键槽位和后续处理建议。

知识库模块：

- 管理售后规则、FAQ、平台政策和异常处理说明。
- 支持关键词检索和分类过滤。
- 为 AI 增强回复提供可追溯的依据文本。

订单上下文模块：

- 准备一批演示用订单样例，包括待发货、已签收、退货中、退款中、物流异常等状态。
- 根据用户提供的订单号或当前会话绑定订单，输出可用于业务判断的上下文。

回复生成模块：

- 优先执行业务规则判断，确定能否退换、需要什么材料、下一步如何处理。
- 命中知识后组织依据文本。
- AI 可用时由 LangChain4j 生成自然语言回复；AI 不可用时使用本地模板兜底。

日志与追踪模块：

- 记录每次用户提问、意图识别结果、知识命中、订单上下文、AI 调用状态和最终回答。
- 为演示和排错提供可查证的处理链路。

## 5. 接口与数据规划

### 5.1 主要接口

详细请求参数、响应结构、错误码和联调顺序以 `docs/backend-api-design.md` 为准。总计划中只保留接口分组清单。

聊天接口：

- `POST /api/chat/sessions`：创建会话。
- `GET /api/chat/sessions`：查询会话列表。
- `GET /api/chat/sessions/{sessionId}`：查询会话详情。
- `POST /api/chat/sessions/{sessionId}/messages`：发送用户消息并返回系统回复。

知识库接口：

- `GET /api/knowledge/docs`：查询知识文档列表。
- `POST /api/knowledge/docs`：新增知识文档。
- `PUT /api/knowledge/docs/{id}`：更新知识文档。
- `DELETE /api/knowledge/docs/{id}`：删除知识文档。
- `GET /api/knowledge/search?query=`：检索知识库。

订单与系统接口：

- `GET /api/orders/{orderNo}`：查询演示订单上下文。
- `GET /api/system/status`：查询后端、数据库、AI 增强层状态。
- `GET /api/logs/chat/{sessionId}`：查询会话处理轨迹。

### 5.2 数据表规划

详细字段、表关系、索引、枚举和初始化数据以 `docs/database-design.md` 为准。总计划中只保留核心表清单。

建议先建设以下核心表：

- `knowledge_doc`：知识文档表，保存标题、分类、场景、内容、状态、更新时间。
- `demo_order`：演示订单表，保存订单号、商品、金额、订单状态、物流状态、售后状态。
- `chat_session`：会话表，保存会话编号、用户标识、绑定订单、创建时间、更新时间。
- `chat_message`：消息表，保存用户消息、系统回复、角色、所属会话、时间。
- `intent_record`：意图识别记录表，保存意图标签、置信度、槽位信息。
- `retrieval_log`：知识检索日志表，保存查询词、命中文档、相似度或命中原因。
- `ai_call_log`：AI 调用日志表，保存模型名称、请求状态、耗时、异常摘要。

开发初期可以只做单管理员模式，权限表可放到后续增强阶段。

## 6. 实施阶段安排

### 阶段一：工程初始化

- 新建 `server/` Spring Boot 工程，配置 Web、Validation、MySQL、MyBatis Plus 或 JPA、LangChain4j 依赖。
- 新建 `web/` Vue 3 工程，配置 Vite、Vue Router、Pinia、Axios、Element Plus。
- 新建 `sql/schema.sql` 和 `sql/seed.sql`，准备数据库结构和初始演示数据。

阶段产出：后端健康检查接口、前端空页面、数据库可连接。

### 阶段二：基础业务闭环

- 实现会话创建、消息发送、消息保存和历史查询。
- 实现知识库 CRUD 与基础检索。
- 实现演示订单查询和订单上下文返回。
- 前端完成咨询工作台和知识库列表的基础交互。

阶段产出：不依赖 AI 也能完成规则问答和知识依据展示。

### 阶段三：售后意图与规则回复

- 完成 7 类售后意图的规则识别。
- 建立不同意图对应的处理模板和追问策略。
- 接入订单状态判断，例如是否签收、是否超时、是否退款中。
- 前端展示意图标签、订单信息、命中规则和处理建议。

阶段产出：覆盖主要售后场景的稳定演示链路。

### 阶段四：LangChain4j AI 增强

- 封装 LangChain4j 调用服务，统一管理模型配置和异常处理。
- 设计提示词模板，把用户问题、会话历史、订单上下文、知识命中结果装配到提示词中。
- 实现 AI 回复和本地模板回复的双模式切换。
- 记录 AI 调用日志，前端显示 AI 是否参与本次回答。

阶段产出：回答更自然，同时具备 AI 异常兜底能力。

### 阶段五：前端完善与演示材料

- 优化咨询工作台布局，让聊天、依据、订单、轨迹展示清晰。
- 增加预置演示案例和一键填充测试问题。
- 补充系统状态页和日志查看页。
- 整理测试用例、演示脚本、系统截图和结项说明材料。

阶段产出：可用于课堂答辩和结项验收的完整演示版本。

## 7. 测试计划

功能测试：

- 会话创建、消息发送、历史查询。
- 知识文档新增、编辑、删除、检索。
- 订单号查询和订单上下文展示。
- 7 类售后意图识别。
- AI 可用与 AI 不可用两种回复模式。

场景测试：

- 用户询问“这个订单能不能退货”。
- 用户追问“那多久到账”。
- 用户询问“物流一直不动怎么办”。
- 用户询问“超过七天还能换吗”。
- 用户表达投诉和人工客服转接诉求。

稳定性测试：

- MySQL 未连接时返回明确错误。
- AI 配置缺失时自动切换本地模板。
- 知识库未命中时给出补充信息提示。
- 用户没有提供订单号时引导补充订单信息。

## 8. 风险与应对

- AI 调用不稳定：保留本地规则模板，保证演示时仍可正常回复。
- 功能范围过大：优先完成咨询工作台、知识库、会话、订单上下文和回复生成主链路，权限、多角色、复杂统计放到后续优化。
- 知识检索效果不足：先采用关键词和分类过滤保证可控，再逐步补充向量检索或重排序。
- 前后端联调耗时：接口先固定返回结构，前端可用 mock 数据并行开发。
- 答辩展示压力：提前准备固定演示脚本和内置案例，减少现场输入的不确定性。
